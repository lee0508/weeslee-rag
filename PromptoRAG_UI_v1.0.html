입력 파라미터 — 변수형 템플릿 구조
프롬프트를 매번 직접 수정하지 않고, 변수만 입력해 재사용합니다

이 시스템의 핵심은 프롬프트를 "고정 문장"이 아니라 "변수형 템플릿"으로 관리하는 것입니다. 사용자는 변수 값만 입력하고, 나머지는 템플릿이 자동으로 구성합니다.

📌 예시
기관명 = 과학기술정보통신부
사업명 = 행정업무 AX 정보화전략계획 수립
작성목적 = ISP 수립
출력형식 = 표 + 서술 혼합

→ AI가 이 변수들을 조합해 완성된 프롬프트를 자동 구성한 후 LLM에 전송합니다.
🗄 DB 구현 코드 (변수 정의 테이블)
-- 프롬프트 변수 정의 테이블 CREATE TABLE prompt_variables ( id INT PRIMARY KEY AUTO_INCREMENT, prompt_id INT NOT NULL, -- 연결된 프롬프트 ID var_name VARCHAR(50), -- 변수명: field_agency label VARCHAR(100), -- 표시명: 기관명 input_type ENUM('text','select','number','date'), is_required BOOLEAN DEFAULT TRUE, default_val TEXT, -- 기본값 example_val TEXT, -- 예시값 options JSON, -- select일 때 선택지 sort_order INT DEFAULT 0, FOREIGN KEY (prompt_id) REFERENCES prompts(id) );
💡 API 예시: GET /api/prompts/{id}/variables → 변수 목록 반환 → 프론트에서 동적 폼 렌더링
✏️
추가 요청 입력
이번 실행에만 적용되는 임시 지시사항

템플릿 변수 외에, 이번 실행에만 필요한 특수한 요청을 자유 형식으로 입력하는 공간입니다.

📌 활용 예시
  • "ODA 제안서 문체로 작성해 주세요"
  • "위험요인을 별도 박스로 강조해 주세요"
  • "3페이지 분량으로 맞춰 주세요"
  • "이해관계자 분석을 표로 정리해 주세요"
  • "임원보고용 3줄 요약을 문서 맨 앞에 추가해 주세요"
🗄 DB 구현 (실행 이력에 함께 저장)
-- 실행 이력 테이블에 extra_request 컬럼 포함 CREATE TABLE execution_logs ( id INT PRIMARY KEY AUTO_INCREMENT, user_id INT NOT NULL, prompt_id INT NOT NULL, prompt_version VARCHAR(10), input_vars JSON, -- 변수 입력값 전체 extra_request TEXT, -- 추가 요청 내용 ks_ids JSON, -- 선택된 지식원 ID 목록 model_used VARCHAR(50), result_text LONGTEXT, -- 생성 결과 rating TINYINT, -- 1-5점 만족도 tokens_used INT, created_at DATETIME DEFAULT CURRENT_TIMESTAMP );

AI 생성 결과 — 후처리 기능
결과를 단순 복사로 끝내지 않고, 자산으로 축적합니다
후처리 기능 목록
  • 다시 생성: 동일 입력으로 결과를 재생성합니다
  • 부분 수정 요청: 특정 문단만 선택하여 수정을 요청합니다
  • 표로 변환: 서술형 내용을 구조화된 표로 변환합니다
  • 내보내기: Word/HWP/PDF/메일 형식으로 출력합니다
  • 결과 저장: 결과 보관함에 저장하여 재활용합니다
  • 템플릿 개선 요청: 관리자에게 템플릿 개선을 요청합니다
🗄 DB 구현 (결과 보관함)
-- 결과 보관함 테이블 CREATE TABLE result_archive ( id INT PRIMARY KEY AUTO_INCREMENT, exec_log_id INT, -- 연결된 실행 이력 ID user_id INT, title VARCHAR(200), -- 사용자가 지정한 제목 content LONGTEXT, -- 저장된 결과 본문 format VARCHAR(20), -- 'html', 'markdown', 'plain' tags JSON, -- 태그 목록 is_shared BOOLEAN, -- 부서 공유 여부 created_at DATETIME, updated_at DATETIME );
💡 내보내기 구현: POST /api/export → {format: 'docx'|'hwpx'|'pdf'} → 서버에서 변환 후 다운로드 URL 반환

지식원 선택 — RAG 데이터 범위 설정
어떤 문서를 근거로 답변을 생성할지 선택합니다

지식원은 LLM이 답변을 생성할 때 참조하는 문서 저장소입니다. 선택한 지식원에서만 근거를 찾아 답변하므로, 목적에 맞는 문서를 선택하는 것이 중요합니다.

📌 지식원 유형별 활용 예시
ISP 계획서 작성 → 과기정통부 사업 문서 + 정책/법령 문서 선택
법령 질의응답 → 정책/법령 문서만 선택
내부 회의 정리 → 부서 공유 문서 + 개인 업로드 선택
ODA 제안서 → 전체 문서 저장소 선택
🗄 DB 구현 (지식원 관리)
-- 지식원(Knowledge Source) 관리 테이블 CREATE TABLE knowledge_sources ( id INT PRIMARY KEY AUTO_INCREMENT, name VARCHAR(100), ks_type ENUM('global','dept','project','personal','policy'), vector_index VARCHAR(100), -- FAISS / Chroma 인덱스 경로 doc_count INT, access_level TINYINT, -- 1=공개 2=부서 3=임원 4=기밀 owner_id INT, -- 개인 지식원의 소유자 dept_id INT, -- 부서 지식원 소속 부서 updated_at DATETIME ); -- 문서-지식원 매핑 CREATE TABLE ks_documents ( ks_id INT, doc_id INT, chunk_id INT, -- 청크 분할 단위 ID embedding BLOB, -- 임베딩 벡터 (또는 외부 VectorDB 참조) PRIMARY KEY (ks_id, doc_id, chunk_id) );
💡 RAG 구현 스택: LangChain / LlamaIndex + FAISS / ChromaDB + HuggingFace Embeddings (또는 OpenAI/Claude Embedding)
🔧
검색 조건 설정 — RAG 검색 파라미터
검색 정확도와 범위를 제어합니다
파라미터 설명
  • Top-K: 검색 결과 중 LLM에 전달할 문서 청크 수. 높을수록 풍부한 근거, 낮을수록 핵심 집중
  • 관련도 높은순: 질문과 유사도가 높은 청크 우선 반환 (코사인 유사도)
  • 최신 문서 우선: 최신 업로드/수정 문서를 우선 반환
  • 기간 필터: 특정 기간 이내 문서만 검색 범위에 포함
  • 보안 등급: 사용자 권한 내의 문서만 검색 (자동 필터)
🗄 DB 구현 (검색 설정 저장)
-- 사용자별 검색 기본 설정 저장 CREATE TABLE user_search_prefs ( user_id INT PRIMARY KEY, top_k INT DEFAULT 5, sort_mode ENUM('relevance','recency','hybrid'), period_filter INT, -- 개월 수 (NULL=전체) doc_types JSON, -- ['pdf','hwp','xlsx'] updated_at DATETIME ); -- RAG 검색 실행 API 예시 (Python FastAPI) -- POST /api/rag/search -- { query, ks_ids, top_k, sort, period, security_level } -- → { chunks: [{doc_id, chunk, score, doc_title, page}] }
📌
근거 문서 패널
RAG 검색 결과로 찾은 실제 참조 문서를 확인합니다

이 패널이 일반 챗봇과 업무형 RAG UI를 구분하는 가장 핵심적인 요소입니다. 결과보다 근거를 먼저 확인하세요.

📌 표시 정보
  • 참조 문서명: 근거가 된 원본 문서 제목
  • 문서 위치: 페이지 번호, 조항 번호, 문단 위치
  • 발췌 내용: 실제 참조된 텍스트 청크
  • 신뢰도/관련도: 벡터 유사도 점수 (0.0~1.0)
  • 원문 보기: 원본 문서를 새 창으로 열기
🗄 DB 구현 (근거 이력 저장)
-- 근거 이력 테이블 (실행 이력과 연결) CREATE TABLE reference_logs ( id INT PRIMARY KEY AUTO_INCREMENT, exec_log_id INT NOT NULL, -- 실행 이력 ID doc_id INT, -- 원본 문서 ID chunk_id INT, -- 청크 ID chunk_text TEXT, -- 실제 발췌 내용 score FLOAT, -- 유사도 점수 (0.0-1.0) page_num INT, para_num INT, is_applied BOOLEAN, -- 최종 결과에 반영 여부 FOREIGN KEY (exec_log_id) REFERENCES execution_logs(id) );

주의 알림 패널
검색 품질과 문서 상태에 대한 자동 경고
경고 유형
  • 근거 부족: 검색된 관련 문서 수가 너무 적을 때
  • 문서 노후화: 참조 문서가 1년 이상 미갱신 상태일 때
  • 상충 문서: 서로 내용이 충돌하는 문서가 검색될 때
  • 권한 제외: 더 관련성 높은 문서가 있으나 접근 권한 없을 때
  • OCR 처리 중: 업로드 문서가 아직 인덱싱되지 않을 때

과거 실행 결과를 자산으로 활용합니다

실행 이력은 단순 로그가 아니라 재사용 가능한 자산입니다. 과거에 잘 만들어진 결과물의 입력값을 그대로 불러와 다시 실행하거나, 결과를 직접 재활용할 수 있습니다.

활용 방법
  • 입력값 재사용: 과거 실행의 변수 설정을 그대로 불러옵니다
  • 결과 재활용: 저장된 결과를 편집하여 사용합니다
  • 비교 검토: 동일 입력의 다른 버전 결과를 비교합니다
