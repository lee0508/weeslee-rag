Step 1: Source Scan

원본 폴더를 스캔하여 전체 데이터셋의 기준 구조를 만듭니다.

나무 구조 기준

  • 뿌리: snapshot_id
  • 큰 줄기: source_id (01. RFP, 02. 제안서, 03. 산출물)
  • 작은 줄기: category_id (하위 폴더)
  • 잎: document_id (파일별 고유 ID)

처리 내용

  • 00. RAG 소스 폴더 스캔
  • source_id, category_id, document_id 생성
  • documents.jsonl 생성
  • snapshot 정보 기록

출력 결과

전체 파일 수
-
source별 파일 수
-
document_id 생성 수
-

Step 8: Graph Build (Ontology / JSON Graph)

Step 10: Search Quality / Activate

운영 반영 전에 검색 품질을 검증하고, 검증 완료된 Snapshot만 활성화합니다.

주의사항: Search Quality 검증 없이 Activate하지 않습니다. 기존 운영 Index를 직접 덮어쓰지 말고 Snapshot pointer 방식으로 활성화합니다.

Search Quality 검증

테스트 질문별 Top-K 검색 결과, source_id/category_id 필터 동작, RFP→제안서 유사 검색, GraphRAG 관계 검색, Wiki 검색 품질 검사

Activate 설정

Search Quality 검증 완료된 Snapshot만 선택 가능
이전 Snapshot 정보 저장 (rollback 가능)

Execution APIs

POST /api/admin/search-quality/run POST /api/admin/dataset/activate POST /api/admin/dataset/rollback

Step 1 OCR/파싱 설정

PDF, HWP, HWPX, DOCX, PPTX, XLSX 파일에서 텍스트를 추출합니다. 이미지 기반 PDF는 OCR을 사용합니다.
이미지 PDF 또는 스캔 문서에 OCR 적용
높을수록 정확하지만 처리 시간 증가
추출 텍스트가 이보다 짧으면 OCR fallback
표를 마크다운 형식으로 변환
📋 Request Body 미리보기 ▼ 
{
  "ocr_mode": "auto",
  "ocr_language": "kor+eng",
  "ocr_dpi": 300,
  "ocr_engine": "tesseract",
  "min_text_length": 50,
  "image_preprocess": "none",
  "hwp_extractor": "pyhwp",
  "table_extract": true
}

Step 2 청킹/임베딩 설정

권장: 500~2000 토큰
권장: chunk 크기의 10~20%
📋 Request Body 미리보기 ▼ 
{
  "chunk_size": 1000,
  "chunk_overlap": 200,
  "embedding_model": "nomic-embed-text",
  "embedding_provider": "ollama"
}
