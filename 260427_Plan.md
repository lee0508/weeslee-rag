# 위즐리앤컴퍼니 문서 중앙화 시스템 개발 계획

**작성일:** 2026-04-27  
**문서명:** `260427_Plan.md`  
**대상 조직:** 위즐리앤컴퍼니  
**목표:** 15년 누적 컨설팅 문서(PDF, PPT/PPTX, DOC/DOCX 등)를 중앙화하고, 신규 과업지시서/수행기관 정보/입찰문서와 비교하여 과거 연관 문서를 신속하게 찾아주는 시스템 구축

---

## 1. 추진 배경

위즐리앤컴퍼니는 장기간 축적된 제안서, 착수보고서, 중간/최종보고서, 입찰문서, 과업지시서, 발표자료, 수행 실적 문서를 보유하고 있다.  
현재 문서는 파일 서버, 개인 PC, 메일 첨부, 프로젝트 폴더 등으로 분산되어 있을 가능성이 높고, 동일하거나 유사한 과업을 다시 수행할 때 다음 문제가 반복된다.

1. 필요한 과거 문서를 찾는 데 시간이 오래 걸린다.
2. 같은 기관, 같은 사업유형, 비슷한 예산/범위의 과거 산출물을 재활용하기 어렵다.
3. 문서 간 관계가 파일명 수준에 머물러 있어 "왜 이 문서가 관련 있는지" 설명하기 어렵다.
4. 문서 최신성, 버전, 수행기관, 사업분야, 문서유형 메타데이터가 일관되게 관리되지 않는다.
5. 검색은 되더라도 신규 문서와 과거 문서의 구조적 비교가 어렵다.

따라서 본 시스템은 단순 문서 저장소가 아니라, 문서 텍스트와 메타데이터, 문서 간 관계, 버전 이력을 함께 관리하는 지식기반형 문서 운영체계로 설계해야 한다.

## 2. 목표 정의

### 2.1 비즈니스 목표

1. 신규 사업 검토 시 과거 유사 실적과 산출물을 5분 이내에 탐색 가능하게 한다.
2. 입찰/제안/수행 단계에서 재사용 가능한 문서 후보를 자동 추천한다.
3. 기관, 사업유형, 문서유형, 수행연도, 키워드, 목차구성, 요구사항 기준으로 비교 탐색이 가능해야 한다.
4. 문서 자산을 개인 의존형이 아닌 회사 공용 지식자산으로 전환한다.

### 2.2 시스템 목표

1. 원본 문서 중앙 수집 및 표준 메타데이터 관리
2. OCR/텍스트 추출/구조화/청킹 자동화
3. 벡터 검색 + 지식 그래프 + 규칙 기반 필터 결합 검색
4. "왜 추천되었는지" 설명 가능한 결과 제공
5. 문서 개정 및 파생문서 이력 관리
6. 향후 초안 작성 보조와 요약, 비교분석 기능으로 확장 가능한 구조 확보

## 3. 참조 자료에서 도출한 핵심 설계 원칙

### 3.1 Graphify 참조 시사점

Graphify 아키텍처는 문서를 단순 청크 집합으로 보지 않고, 추출(`extract`) 후 그래프 구축(`build`), 군집화(`cluster`), 분석(`analyze`)을 수행하는 다단계 파이프라인을 제시한다.  
특히 다음 개념이 본 과제에 유효하다.

1. 구조는 검색 신호다.  
   문서 간 공통 단어보다 "수행기관-사업유형-산출물-챕터-키워드-근거문장" 연결이 더 중요하다.
2. 멀티모달 통합이 필요하다.  
   PDF, PPT, Word, 표, 이미지 기반 도식까지 함께 다뤄야 한다.
3. provenance가 중요하다.  
   추천 결과의 근거가 추출값인지, 추론값인지, 사람이 보정한 값인지 구분되어야 한다.
4. 그래프는 토큰 비용을 줄인다.  
   전체 문서를 매번 다시 읽는 대신 관계 그래프와 요약 노드로 질의 비용을 줄일 수 있다.

### 3.2 Karpathy의 LLM Wiki 참조 시사점

Karpathy의 `LLM Wiki` 아이디어는 질의 시마다 원문을 재탐색하는 전통적 RAG 대신, 지속적으로 축적되고 갱신되는 중간 산출물로서의 위키를 강조한다.  
이 관점은 위즐리앤컴퍼니에 매우 적합하다.

1. 지식은 매 질의마다 재발견하지 말고 축적되어야 한다.
2. 문서를 읽고 나면 엔티티 페이지, 사업유형 페이지, 기관 페이지, 방법론 페이지가 갱신되어야 한다.
3. 상충 정보, 오래된 정보, 재사용 가능한 템플릿 포인트를 문서화해야 한다.

즉, 본 시스템은 `원문 저장소 + 검색 인덱스`만이 아니라 `운영형 지식 위키`를 함께 가져가야 한다.

### 3.3 legalize-kr 참조 시사점

`legalize-kr`는 문서를 Markdown과 Git commit 이력으로 관리하며 시점별 비교와 변경 추적이 가능하다는 점이 핵심이다.  
이를 본 과제에 적용하면 다음 원칙이 도출된다.

1. 원문 파일은 원본 그대로 보관한다.
2. 추출 텍스트/정규화 Markdown/메타데이터 YAML은 Git 친화적으로 관리한다.
3. 문서 개정, 파생본, 기관별 변형본의 이력을 추적 가능하게 한다.
4. 특정 시점 기준으로 문서 상태를 재현할 수 있어야 한다.

## 4. 권장 목표 아키텍처

본 시스템은 다음 5계층으로 설계한다.

### 4.1 원천 수집 계층

대상:

1. 사내 파일 서버
2. 프로젝트별 공유 폴더
3. 관리자 수동 업로드
4. 향후 메일/그룹웨어/드라이브 연계

수집 시 저장 항목:

1. 원본 파일
2. 파일 해시
3. 파일 경로
4. 생성일/수정일
5. 프로젝트명 추정값
6. 수행기관 추정값
7. 문서유형 추정값

### 4.2 문서 처리 계층

1. 파일 형식 판별
2. 텍스트 추출
   - PDF: 디지털 PDF 우선, 스캔본은 OCR
   - PPT/PPTX: 슬라이드 단위 추출
   - DOC/DOCX: 문단/표/제목 구조 추출
3. 문서 구조화
   - 제목, 목차, 장/절, 표, 핵심 키워드, 기관명, 사업명, 연도, 예산, 기간, 방법론
4. 품질 검수
   - 추출 실패, OCR 품질, 빈 문서, 중복 문서 탐지

### 4.3 지식화 계층

이 계층이 본 과제의 핵심 차별화 영역이다.

1. 벡터 인덱스 구축
   - 본문 청크 임베딩
   - 문서 요약 임베딩
   - 장/절 단위 임베딩
2. 메타데이터 인덱스 구축
   - 수행기관, 사업유형, 문서유형, 산업군, 작성연도, 지역, 주제태그
3. 관계 그래프 구축
   - `문서 -> 프로젝트`
   - `문서 -> 수행기관`
   - `문서 -> 사업유형`
   - `문서 -> 문서유형`
   - `문서 -> 키워드`
   - `문서 -> 챕터`
   - `신규 과업지시서 -> 요구사항`
   - `요구사항 -> 관련 과거 문서`
4. 운영형 위키 생성
   - 기관 페이지
   - 사업유형 페이지
   - 방법론/프레임워크 페이지
   - 자주 재사용되는 목차/표현/산출물 패턴 페이지

### 4.4 검색/추천 계층

검색은 단일 방식이 아니라 하이브리드로 설계한다.

1. 키워드 검색
2. 메타데이터 필터 검색
3. 벡터 유사도 검색
4. 그래프 기반 확장 검색
5. 재순위화(reranking)
6. 설명 생성

추천 시나리오:

1. 신규 과업지시서를 업로드하면
2. 요구사항/기관/사업유형/산출물 유형을 추출하고
3. 과거 문서를 후보군으로 찾은 뒤
4. 관련도 점수와 근거를 함께 보여준다.

### 4.5 운영/관리 계층

1. 관리자 대시보드
2. 수집 현황 모니터링
3. 추출 오류 큐
4. 메타데이터 보정 화면
5. 문서 승인/비공개 권한 관리
6. 감사로그 및 사용자 검색 로그

## 6. 권장 데이터 모델

### 6.1 주요 엔티티

1. `Document`
2. `DocumentVersion`
3. `Project`
4. `Organization`
5. `DocumentType`
6. `TaskRequirement`
7. `Section`
8. `Keyword`
9. `Attachment`
10. `KnowledgePage`

### 6.2 주요 관계

1. 문서는 하나 이상의 프로젝트에 연결된다.
2. 프로젝트는 하나 이상의 수행기관/발주기관과 연결된다.
3. 문서는 문서유형을 가진다.
4. 문서는 여러 섹션과 청크를 가진다.
5. 청크와 섹션은 키워드, 요구사항, 위키 페이지와 연결된다.
6. 문서 버전은 이전 버전과 연결된다.
7. 추천 결과는 근거 청크와 근거 관계를 반드시 가진다.

### 6.3 메타데이터 표준안

1. 문서 ID

3. 정규화 파일명
4. 문서유형
5. 프로젝트명
6. 수행기관/발주기관
7. 작성연도
8. 사업분야
9. 보안등급
10. 버전
11. 언어
12. 추출상태
13. OCR 여부
14. 해시값
15. 원본 저장경로

## 7. 권장 기술 스택

현재 저장소 구조(`backend`, `frontend`)를 고려하면 다음 구성이 현실적이다.

### 7.1 백엔드

1. FastAPI
2. SQLAlchemy + Alembic
3. Celery 또는 RQ 기반 비동기 작업
4. Redis 캐시/작업 큐

### 7.2 프론트엔드

1. Next.js 또는 현재 프론트엔드 스택 유지
2. 관리자용 문서 검수/검색 UI
3. 결과 비교 패널 UI

### 7.3 저장소

1. 원본 파일 저장소: NAS 또는 사내 파일 서버
2. 관계형 DB: MySQL 또는 PostgreSQL
3. 벡터 저장소: pgvector, Qdrant, 또는 FAISS 기반 초기 구축
4. 그래프 저장소: 초기에는 RDB + NetworkX/JSON 그래프, 확장 시 Neo4j 검토
5. Git 저장소: 정규화 Markdown/YAML/위키/버전 이력 관리

### 7.4 문서 처리

1. PDF: `pdfplumber`, `pymupdf`, OCR 필요 시 `tesseract`
2. PPTX: `python-pptx`
3. DOCX: `python-docx`
4. 스캔본 OCR: `ocrmypdf` 또는 Tesseract 파이프라인

### 7.5 AI/검색

1. 한국어 강한 임베딩 모델 선정
2. reranker 도입
3. LLM 기반 메타데이터 추출 및 요약
4. 그래프 관계 추론은 낮은 신뢰도 태그와 함께 저장

## 8. 검색/추천 로직 설계

### 8.1 입력

1. 신규 과업지시서
2. 입찰공고문
3. 수행기관 제안요청서
4. 사용자의 자연어 검색 질의

### 8.2 처리 절차

1. 문서에서 기관명, 사업유형, 핵심 요구사항, 산출물, 일정, 예산 키워드 추출
2. 메타데이터 필터로 1차 후보군 축소
3. 벡터 검색으로 유사 청크/문서 탐색
4. 그래프 탐색으로 연관 프로젝트와 파생 문서 확장
5. reranker로 상위 결과 재정렬
6. LLM이 추천 사유와 활용 포인트 요약

### 8.3 결과 화면 예시

1. 상위 유사 문서 10건
2. 관련 프로젝트 5건
3. 동일 기관 과거 실적
4. 유사한 목차 패턴
5. 재사용 가능한 챕터 후보
6. 추천 근거 문장 및 점수

### 8.4 점수 구성 예시

1. 메타데이터 일치도 25%
2. 벡터 유사도 35%
3. 그래프 연관도 20%
4. 문서 최신성 10%
5. 사용자 피드백/클릭 신호 10%

## 9. 저장 방식 원칙

### 9.1 원본 파일

원본은 수정하지 않고 별도 보관한다.

### 9.2 정규화 산출물

문서별로 다음 산출물을 생성한다.

1. 추출 텍스트 `.txt`
2. 구조화 Markdown `.md`
3. 메타데이터 `.yaml` 또는 `.json`
4. 문서 요약 `.md`
5. 청크 데이터 `.jsonl`

### 9.3 Git 관리 대상

Git에는 다음을 우선 관리한다.

1. 정규화 Markdown
2. 메타데이터
3. 위키 페이지
4. 규칙/사전/분류체계
5. 버전 비교 기록

이 방식은 `legalize-kr`처럼 변경 이력과 시점 비교를 가능하게 하며, 원문 자체는 대용량 저장소 정책에 맞게 별도 저장한다.

---

## 10. 단계별 개발 계획

### 10.1 1단계: 진단 및 표준화(2~3주)

1. 보유 문서 저장 위치 조사
2. 파일 유형, 수량, 용량, 연도별 분포 파악
3. 문서유형 분류체계 수립
4. 필수 메타데이터 표준 확정
5. 보안등급/접근권한 정책 정의

산출물:

1. 문서 자산 인벤토리
2. 메타데이터 표준안
3. 문서 분류체계
4. 파일 수집 정책

### 10.2 2단계: 수집/추출 파이프라인 구축(3~5주)

1. 문서 등록 API
2. 파일 저장 전략 구현
3. 형식별 텍스트 추출기 구축
4. OCR 예외 처리
5. 중복 문서 감지
6. 추출 결과 검수 화면

산출물:

1. ingest pipeline
2. extraction pipeline
3. 품질 검수 큐

### 10.3 3단계: 검색 MVP 구축(4~6주)

1. 청킹/임베딩/벡터 인덱스 구축
2. 메타데이터 검색 구현
3. 하이브리드 검색 API 구현
4. 추천 근거 문장 노출
5. 검색 UI 구현

산출물:

1. 문서 검색 MVP
2. 신규 문서 기반 유사 문서 추천 기능

### 10.4 4단계: 그래프/위키 계층 구축(4~6주)

1. 엔티티 추출
2. 문서 관계 그래프 구축
3. 프로젝트/기관/방법론 위키 자동 생성
4. 그래프 기반 탐색 UI
5. 추천 설명력 강화

산출물:

1. knowledge graph
2. knowledge wiki
3. explainable recommendation

### 10.5 5단계: 운영 안정화 및 고도화(지속)

1. 피드백 기반 랭킹 개선
2. 문서 버전 비교
3. 초안 작성 보조
4. 사용 로그 기반 추천 개선
5. 권한체계 고도화

## 11. 우선순위 제안

가장 중요한 것은 "모든 문서를 완벽히 이해하는 AI"가 아니라, "실무자가 바로 쓸 수 있는 재현성 있는 검색 체계"를 먼저 만드는 것이다.  
따라서 우선순위는 다음과 같이 권장한다.

1. 수집 표준화
2. 메타데이터 정비
3. 추출 품질 확보
4. 하이브리드 검색 MVP
5. 그래프 및 위키 고도화
6. 생성형 기능 확장

---

## 12. 주요 리스크와 대응

### 12.1 리스크

1. 스캔본/OCR 품질 저하
2. 파일명만으로는 프로젝트/기관 식별 어려움
3. 중복 문서와 최종본/수정본 혼재
4. 보안 문서 접근통제 필요
5. 초기 메타데이터 정제 비용 큼

### 12.2 대응

1. OCR 신뢰도 점수와 수동 검수 프로세스 도입
2. 자동 추출 후 관리자 보정 워크플로우 제공
3. 파일 해시 + 유사도 + 버전 규칙으로 중복 판정
4. 권한 기반 검색 결과 마스킹
5. 우선 사업분야 또는 최근 3~5년 문서부터 단계 도입

---

## 13. 성공 지표(KPI)

1. 검색 성공률: 사용자가 원하는 연관 문서를 3회 이내 검색으로 찾는 비율
2. 추천 정확도: 상위 10개 추천 중 실제 유효 문서 비율
3. 메타데이터 완성도: 필수 메타데이터 자동/수동 확보율
4. 문서 처리 성공률: 텍스트 추출 성공 비율
5. 재사용 시간 절감: 신규 제안/수행 준비 시간 단축률

---

## 14. 즉시 실행 권고안

현 시점에서 위즐리앤컴퍼니가 바로 착수할 실무 순서는 다음과 같다.

1. 문서 표본 300~500건을 선정한다.
2. 문서유형 5종 정도로 제한해 PoC를 수행한다.
   - 과업지시서
   - 제안서
   - 착수보고서
   - 최종보고서
   - 발표자료
3. 수행기관/사업유형/연도 메타데이터를 최소 기준으로 정한다.
4. 하이브리드 검색 MVP를 먼저 구축한다.
5. 이후 그래프와 위키 계층을 추가한다.

이 접근은 실패 비용을 낮추면서도, 실제 사용성이 있는 결과를 빠르게 검증할 수 있다.

## 15. 결론

위즐리앤컴퍼니의 문서 중앙화 시스템은 단순 DMS(Document Management System)나 전통적 RAG 검색기로 설계하면 한계가 명확하다.  
본 과제는 다음 세 축을 동시에 가져가야 한다.

1. `중앙 문서 저장과 표준 메타데이터`
2. `하이브리드 검색(메타데이터 + 벡터 + 그래프)`
3. `지속적으로 축적되는 운영형 지식 위키와 버전 이력`

즉, 최종 목표는 "파일을 찾는 시스템"이 아니라, "회사의 수행 지식을 누적하고 재사용하는 시스템"이어야 한다.  
개발은 MVP부터 단계적으로 진행하되, 아키텍처는 처음부터 그래프/위키/이력 관리로 확장 가능하게 설계하는 것이 바람직하다.
