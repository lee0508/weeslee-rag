# 260507 회사 서버 서비스별 통합 URL 설계안

## 1. 문서 목적

본 문서는 회사 서버 `192.168.0.207`의 현재 서비스 노출 구조를 기준으로, 회사 내부 및 외부 사용자가 보다 일관된 URL 체계로 서비스에 접근할 수 있도록 통합 URL 설계 방향을 정리하기 위한 문서이다.

이번 설계의 핵심 전제는 다음과 같다.

- `80` 포트는 회사 문서 저장소 NASS에서 사용 중이므로 절대 수정하지 않는다.
- 서비스 통합 진입점은 `443` 기반 경로 방식으로 설계한다.
- 기존 개별 포트 서비스는 운영 안정성을 위해 당분간 병행 유지할 수 있다.
- 실제 공개 경로 기준은 프로젝트 폴더명이 아니라 Nginx의 `location` 경로이다.
- 단, 이번 정리 방향은 `oasis`를 제외하고 프로젝트 폴더명을 내부/외부 기본 서비스 경로로 맞추는 것을 원칙으로 한다.

## 2. 현재 운영 원칙

현재 서버는 주로 다음 두 가지 방식으로 서비스가 노출되어 있다.

1. 개별 포트 기반 접근
- `http://192.168.0.207:8096/weeslee-figma/`
- `http://192.168.0.207:8093/KDART/`
- `http://192.168.0.207:8095/tajikstan-enrollment-system-v1/`

2. `443` 기반 경로 접근
- `https://server.weeslee.co.kr/pdf-ocr/`
- `https://server.weeslee.co.kr/KDART/`
- `https://server.weeslee.co.kr/weeslee-rag/`

현재 상태를 고려하면, 향후 표준 운영 방식은 `443 + 서비스 경로`로 통일하는 것이 가장 적절하다.

## 3. 설계 결론

### 3.1 결론

회사 내부 및 외부 공용 접속 주소는 아래 원칙으로 통합한다.

- 내부 기본 URL 형식: `http://192.168.0.207/{project-folder-name}/`
- 외부 기본 URL 형식: `https://server.weeslee.co.kr/{project-folder-name}/`
- `oasis`는 예외 서비스로 별도 유지
- 기존 개별 포트 URL은 이행 기간 동안만 운영 점검용으로 유지
- `80` 포트는 NASS 전용으로 유지하며 서비스 통합 대상으로 사용하지 않음
- 주소 입력 시 프로젝트 폴더명은 영문 대소문자를 구분하지 않도록 처리

### 3.2 이유

- 내부와 외부 URL 규칙을 동일하게 맞출 수 있다.
- 프로젝트 식별 기준을 폴더명으로 통일할 수 있다.
- 외부 공유기 포트 개방 수를 단계적으로 줄일 수 있다.
- 사용자 입장에서 접속 주소가 단순해진다.
- 서비스 추가 시 포트 중심이 아니라 경로 중심으로 관리할 수 있다.
- SSL 인증서, 프록시, 접근제어, 로깅을 `443`에서 일괄 관리하기 쉽다.

## 4. 통합 URL 네이밍 원칙

서비스 경로는 아래 기준으로 정한다.

- 기본적으로 프로젝트 폴더명을 그대로 사용한다.
- 대소문자 정책은 사용자 입력에서는 비구분으로 처리하고, 내부 canonical 경로는 프로젝트 기준 표기값으로 유지한다.
- 경로명은 실제 폴더명과 동일하게 맞춘다.
- 신규 서비스는 가능하면 소문자 또는 하이픈 기준으로 통일하는 것이 바람직하다.

권장 예시:

- `/weeslee-figma/`
- `/weeslee-rag/`
- `/pdf-ocr/`
- `/sdg/`
- `/seniwork/`

기존 호환 유지가 필요한 경우:

- `/KDART/`는 실제 폴더명이 대문자이므로 그대로 유지
- 단, 사용자 입력은 `/kdart/`, `/Kdart/`, `/KDART/` 모두 허용하는 방향으로 처리

## 6. 서비스 유형별 권장 노출 방식

### 6.1 정적 프런트엔드

정적 빌드 결과물 기반 서비스는 `443` 경로 기반 통합에 가장 적합하다.

대상 예시:

- `weeslee-figma`
- `sdg`
- `tajik-task-frontend2`
- `tajik-task-frontend3`

권장 방식:

- `location ^~ /service-name/ { alias ... }`
- SPA인 경우 `try_files $uri $uri/ /service-name/index.html;`

### 6.2 백엔드 또는 SSR 서비스

백엔드나 SSR 기반 앱은 Nginx에서 경로 프록시를 사용한다.

대상 예시:

- `weeslee-rag`
- `KDART`
- `drug-system`
- `kc-monitor`

권장 방식:

- `location ^~ /service-name/ { proxy_pass http://127.0.0.1:port; }`
- API rewrite가 필요한 경우 별도 `/service-name/api/` 블록을 둔다.

### 6.3 혼합형 서비스

정적 프런트와 API가 함께 있는 서비스는 정적 경로와 API 프록시를 분리한다.

대상 예시:

- `tajik-task-frontend2`
- `tajik-task-frontend3`
- `tajikstan-enrollment-system-v1`

권장 방식:

- 프런트: `alias`
- API: `proxy_pass`
- API 경로는 `/service-name/api/` 같은 규칙으로 고정

## 8. 주의사항

### 8.1 현재 Nginx 설정 구조상 주의점

현재 서버는 `nginx.conf`에서 아래 패턴을 포함하고 있다.

- `/etc/nginx/conf.d/*.conf`
- `/etc/nginx/sites-enabled/*`

이 구조에서는 `sites-enabled` 아래 `default.bak_*` 파일도 함께 로드될 수 있다. 실제로 `9283` 포트가 백업 파일에 의해 활성화된 흔적이 있으므로, 향후 설정 정리 시 반드시 점검이 필요하다.

### 8.2 URL 경로와 실제 폴더 경로는 다를 수 있음

예를 들어 아래와 같은 구조가 가능하다.

- 외부 URL: `/weeslee-rag/`
- 실제 앱: `127.0.0.1:8080`

따라서 서비스 통합 설계는 "폴더명 기준"이 아니라 "사용자에게 보여줄 서비스 경로 기준"으로 정해야 한다.

### 8.3 프런트엔드 서브경로 대응

통합 URL 구조로 옮길 때 프런트엔드 앱은 아래 항목을 확인해야 한다.

- 정적 asset 경로
- router basename
- API base URL
- 절대경로 하드코딩 여부

이 부분이 맞지 않으면 `443` 경로는 열려도 화면이 깨지거나 API 호출이 실패할 수 있다.

## 9. 권장 실행 순서

1. `443` 경로 기준으로 서비스별 표준 URL을 확정한다.
2. 현재 외부 공지 또는 내부 문서에 사용 중인 포트 URL을 조사한다.
3. 포트 URL과 `443` 통합 URL을 일정 기간 병행 운영한다.
4. 각 서비스의 정적 파일 경로, API 경로, 라우팅을 서브경로 기준으로 점검한다.
5. Nginx 설정에서 중복 포트와 백업 파일 로드 구조를 정리한다.
6. 최종적으로 외부 공개 포트 축소 여부를 검토한다.

## 10. 최종 제안

현재 회사 서버의 통합 URL 기준은 아래와 같이 잡는 것이 적절하다.

- 표준 주소 체계: `https://server.weeslee.co.kr/{service-path}/`
- `80` 포트: NASS 전용, 변경 금지
- 개별 포트 URL: 운영/테스트/이행 기간 동안만 유지
- 신규 서비스: 가능하면 처음부터 `443` 경로 기준으로 설계

특히 회사 홈페이지 개발 서비스는 아래 주소를 표준으로 사용한다.

- 표준 주소: `https://server.weeslee.co.kr/weeslee-figma/`
- 내부 점검 주소: `http://192.168.0.207:8096/weeslee-figma/`

이 원칙으로 정리하면 향후 서비스가 추가되어도 운영 기준을 일관되게 유지할 수 있다.

## 12. Nginx 실제 정리 작업 순서

아래 순서대로 진행하는 것이 가장 안전하다.

### 12.1 1단계: 활성 서비스 기준표 확정

먼저 서비스별로 아래 4가지를 표로 고정한다.

- 공식 URL
- 보조 URL
- 실제 정적 폴더 또는 내부 포트
- 서비스 담당 프로젝트명

이 단계의 목적은 "무엇을 유지하고 무엇을 정리할지"를 먼저 고정하는 것이다.

### 12.2 2단계: 백업 파일 활성 로드 구조 제거

현재 가장 큰 구조적 위험은 다음 설정이다.

- `include /etc/nginx/sites-enabled/*;`

이 패턴은 `default.bak_*` 같은 파일도 함께 읽어버릴 수 있다. 따라서 아래 중 하나로 정리해야 한다.

권장안:

- `sites-enabled`에는 실제 활성 파일만 남김
- 백업 파일은 `sites-archive` 같은 별도 폴더로 이동

대안:

- include 패턴을 `.conf`만 읽도록 재구성

이 단계는 서비스 URL 변경보다 먼저 처리하는 것이 바람직하다.

### 12.3 3단계: 정식 vhost와 임시/백업 설정 분리

정식 서비스는 다음처럼 목적별 파일로 분리하는 것이 좋다.

- `main-443.conf`
- `kdart-8093.conf`
- `weeslee-rag-9284.conf`
- `tajik-task-frontend2-8092.conf`
- `tajik-task-frontend3-8094.conf`
- `tajikstan-enrollment-system-v1-8095.conf`

원칙:

- 정식 운영 파일과 임시 백업 파일을 같은 include 경로에 두지 않음
- 포트 전용 서비스는 파일명만 봐도 목적이 드러나게 유지

### 12.4 4단계: 443 기준 표준 URL 공지 체계 반영

설정 정리 후에는 실제 운영 문서, 안내 링크, 관리자 북마크를 표준 URL 기준으로 바꿔야 한다.

예시:

- 회사 홈페이지: `https://server.weeslee.co.kr/weeslee-figma/`
- KDART: `https://server.weeslee.co.kr/KDART/`
- RAG: `https://server.weeslee.co.kr/weeslee-rag/`

이 단계를 하지 않으면 사용자는 계속 포트 URL을 사용하게 된다.

### 12.5 5단계: 외부 개방 포트 최소화 검토

모든 서비스가 `443` 기준으로 안정화된 이후에만 진행한다.

검토 대상:

- `8092`
- `8093`
- `8094`
- `8095`
- `8096`
- `8099`
- `9282`
- `9283`
- `9284`

단, 아래 조건을 모두 만족할 때만 축소를 검토한다.

- 표준 URL이 충분히 공지됨
- 내부 업무에서 포트 URL 의존이 없어짐
- 헬스체크와 운영 점검 절차가 `443` 기준으로 재정비됨

- `/data/weeslee/KDART`

- `/data/weeslee/weeslee-figma`
- 그 외 나머지 프로젝트도 동일 기준 적용

### 16.2 예외 정책

예외는 현재 `oasis`만 둔다.

- `oasis`는 기존 운영 구조와 별도 정책을 유지
- 나머지 프로젝트는 내부/외부 모두 프로젝트 폴더명 기준 URL로 정리

### 16.3 변경 의미

이번 정책 변경은 다음을 의미한다.

- 기존 `:8093/KDART/`, `:8097/Jupiter-2/`, `:8099/pdf-ocr/` 같은 포트 진입 주소는 보조 주소가 된다.
- 사용자가 안내받는 기본 주소는 무조건 폴더명 기반 내부/외부 URL로 통일한다.
- 신규 프로젝트도 처음부터 이 규칙에 맞춰 Nginx 경로를 설계한다.

### 16.4 대소문자 비구분 처리 원칙

구조 개선 방안을 실제 적용할 때는 아래 원칙을 따른다.

- 외부 주소: `https://server.weeslee.co.kr/{Project Folder}`
- 내부 주소: `http://192.168.0.207/{Project Folder}`
- 주소 입력 시 프로젝트 폴더명은 영문 대소문자를 구분하지 않는다.

예시:

- `/KDART/`, `/kdart/`, `/Kdart/` 모두 허용
- `/Jupiter-2/`, `/jupiter-2/`, `/JUPITER-2/` 모두 허용
- `/weeslee-rag/`, `/Weeslee-Rag/`, `/WEESLEE-RAG/` 모두 허용

권장 구현 방식:

- Nginx에서 case-insensitive 매칭 또는 canonical redirect 처리
- 실제 표준 안내 주소는 한 가지 표기만 사용
- 사용자가 다른 대소문자로 입력해도 표준 경로로 리다이렉트

예시 canonical 정책:

- 안내 주소는 `/KDART/`, `/Jupiter-2/`, `/weeslee-rag/`처럼 프로젝트 기준 표기 사용
- 비표준 대소문자 요청은 같은 리소스의 표준 경로로 `301` 또는 `302` 정리

### 16.5 공유기 80 포트 포워딩 전제

2026-05-07 기준 추가 확인 사항:

- 공유기 설정에서 외부 `80` 포트는 `192.168.0.105:80`으로 포워딩된다.
- 즉 외부 `http://...` 기본 진입은 `192.168.0.207`이 아니라 `192.168.0.105`가 사용 중이다.
- 따라서 `192.168.0.207` 서버의 외부 기본 주소는 `http`가 아니라 `https` 기준으로 설계해야 한다.

정리:

- 외부 기본 주소: `https://server.weeslee.co.kr/{Project Folder}`
- 외부 `http://server.weeslee.co.kr/{Project Folder}`는 `192.168.0.207` 표준 주소로 사용할 수 없음
- 내부 기본 주소: `http://192.168.0.207/{Project Folder}`
- `80` 포트는 외부 기준으로 `192.168.0.105`가 사용 중이므로 `192.168.0.207` 외부 서비스 통합 포트로 사용하지 않음

운영상 의미:

- `192.168.0.207`의 외부 표준 서비스 URL은 `443`만 기준으로 잡는다.
- 내부 사내망에서는 `192.168.0.207` 직접 접근 주소를 별도로 유지할 수 있다.
- 향후 Nginx 구조 개선은 외부 `443` 표준화와 내부 직접 접근 경로 정리를 분리해서 진행한다.
