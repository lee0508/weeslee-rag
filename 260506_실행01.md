제 판단 기준으로는 아래 순서가 가장 안전하고 효과적입니다.

최우선 구현 (반드시 먼저)
1️⃣ max_chunks_per_doc

이걸 가장 먼저 해야 합니다.

이유:

현재 가장 큰 문제는:

특정 대형 문서가 검색 결과를 지배

하는 것입니다.

문서에서도:

DOC-20260506-000004
595 chunk
대부분의 검색을 지배

라고 명확히 분석했습니다.

이 상태에서는:

검색 품질이 좋아 보여도
실제로는 diversity가 낮음

입니다.

왜 가장 먼저 해야 하는가?

왜냐하면 지금은:

검색 정확도

보다 더 중요한 것이:

검색 다양성

이기 때문입니다.

실제 제안서 작성에서는:

같은 문서 5 chunk

보다,

다른 문서 5개

가 훨씬 가치 있습니다.

구현 추천값

현재 추천:

MAX_CHUNKS_PER_DOC = 2

좋습니다.

하지만 실제 추천은:

top_k = 20
max_chunks_per_doc = 2
final_context_docs = 5~8 docs

형태가 좋습니다.

두 번째 우선순위
2️⃣ metadata 정규화

이건 사실상:

LLM-Wiki 진입 준비

입니다.

현재 이미:

document_id
category
source_path

는 있습니다.

하지만 다음이 반드시 필요합니다.

organization
project_name
methodology
document_stage
year
domain
왜 중요한가?

현재는:

semantic similarity

만 존재합니다.

하지만 컨설팅 문서에서는 실제로 중요한 건:

같은 기관
같은 방법론
같은 사업 유형

입니다.

예:

K-water ISP
한국환경공단 ISP
스마트시티 ISP

는 문장 similarity보다:

사업 성격 similarity

가 더 중요합니다.

실제 효과

metadata 정규화가 되면:

final_score =
(similarity * 0.6)
+ (organization_bonus * 0.15)
+ (project_bonus * 0.15)
+ (methodology_bonus * 0.1)

같은 rerank가 가능해집니다.

이 순간부터:

"검색"

이 아니라:

"컨설팅 문서 추천"

이 됩니다.

세 번째 우선순위
3️⃣ category filter

이건 구현은 쉽지만 효과는 큽니다.

특히:

RFP 질문
→ proposal 반환

문제가 줄어듭니다.

현재 구조는:

FAISS
→ 전체 검색
→ 후처리

인데,

추천은:

metadata pre-filter
→ FAISS search

입니다.

왜 post-filter보다 pre-filter가 좋은가?

현재 방식:

top 20 검색
→ proposal 18개
→ rfp만 남기면
→ 결과 부족

가능성이 있습니다.

따라서 더 좋은 구조는:

rfp category subset
→ subset index search

입니다.

실제 추천 구조

추천:

indexes/
├── all/
├── rfp/
├── proposal/
├── final_report/
└── presentation/
네 번째 우선순위
4️⃣ reranking

이건 batch-003 이후 매우 중요합니다.

현재는:

vector similarity only

입니다.

하지만 실제 RAG 품질은:

retrieval
보다
rerank

가 훨씬 중요합니다.

추천 방식

현재 단계에서는:

cross encoder

보다 먼저:

rule-based rerank

예:

if category == query_category:
    score += 0.15

if organization == query_org:
    score += 0.10

if project_similarity:
    score += 0.10
