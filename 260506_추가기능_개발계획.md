### P1 — 즉시 구현 가능 (현재 인프라 활용)

#### 2.1 카테고리 필터 API
**문제**: 현재 RAG 쿼리는 전체 인덱스에서 검색 → RFP 쿼리에 proposal/final_report가 섞여 반환  
**해결**: `/api/rag/query`에 `category` 파라미터 추가, FAISS 검색 후 post-filter

```python
# /api/rag/query 요청 예시
{
  "query": "ISP 사업 제안요청서 요구사항",
  "top_k": 5,
  "category": "rfp"          # 추가: null이면 전체 검색
}
```

**구현 위치**: `backend/app/api/rag.py`, `backend/app/scripts/assemble_rag_response.py`  
**예상 효과**: RFP 카테고리 적중률 75% → 90%+ 개선

---

#### 2.2 최대 청크 수 제한 (max_chunks_per_doc)
**문제**: DOC-20260506-000004 (k-water ISP 최종보고서, 595청크)가 대부분의 쿼리에서 지배적  
**해결**: 동일 문서에서 최대 N개 청크만 허용

```python
# assemble_rag_response.py 내 후처리
MAX_CHUNKS_PER_DOC = 2  # 문서당 최대 2개 청크

def deduplicate_by_doc(chunks, max_per_doc=MAX_CHUNKS_PER_DOC):
    seen = {}
    result = []
    for chunk in sorted(chunks, key=lambda x: x["score"], reverse=True):
        doc_id = chunk["document_id"]
        if seen.get(doc_id, 0) < max_per_doc:
            result.append(chunk)
            seen[doc_id] = seen.get(doc_id, 0) + 1
    return result
```

**예상 효과**: 검색 다양성 증가 (현재 평균 3.2 unique docs → 4~5개)

#### 2.3 metadata 정규화
**문제**: 현재 metadata에 organization, project_name이 없음 → LLM-Wiki entity 구성 불가  
**해결**: 추출 파이프라인에서 폴더명/파일명 파싱으로 구조화된 metadata 추가

```json
{
  "document_id": "DOC-20260506-000001",
  "category": "final_report",
  "organization": "한국수자원공사",
  "project_name": "K-water형 고유플랫폼 구축 전략계획",
  "snapshot": "snapshot_2026-05-06_batch-002-top10-v1",
  "source_path": "...",
  "file_ext": ".pptx",
  "folder_depth2": "20250506. K-water형 고유플랫폼 구축을 위한 전략계획 수립"
}
```

**구현 위치**: `backend/scripts/extract_manifest_batch.py` — `metadata_enricher()` 함수 추가  
**추후 활용**: Project Wiki, Organization Wiki 자동 생성의 기초 데이터

### P2 — 단기 구현 (1~2주, LLM 활용)

#### 2.4 Project Wiki 자동 생성
**개념**: 각 문서의 상위 청크 5개를 LLM에 전달 → 구조화된 프로젝트 위키 페이지 생성  
**출력 위치**: `wiki/projects/{project_slug}.md`

```markdown
# 한국수자원공사 ISP 프로젝트

## 관련 기관
- 한국수자원공사

## 문서 목록
- K-water형 고유플랫폼_구축_전략계획_중간보고 (presentation)
- K-Water형 고유플랫폼 구축을 위한 전략계획 수립 착수보고 (kickoff)

## 핵심 요구사항
- 고유 플랫폼 구축
- 디지털 트랜스포메이션

## 사용 방법론
- ISP
- 전략 기획

## provenance
- EXTRACTED: DOC-20260506-000020 (presentation)
- EXTRACTED: DOC-20260506-000022 (kickoff)
```

**구현 방식**:
1. 인덱싱 완료 후 `build_project_wiki.py` 스크립트 실행
2. Ollama `gemma4:latest`로 구조화된 위키 생성
3. `wiki/projects/` 폴더에 markdown 저장
4. 위키 내용도 FAISS 인덱스에 추가 → 검색 커버리지 확대

## 4. 현재 시스템과의 통합 방식

```
[현재]                    [추가]

  → extract               
  → chunk

  → RAG query             →  + category filter
                          →  + max_chunks_per_doc

                          [신규 파이프라인]
```
