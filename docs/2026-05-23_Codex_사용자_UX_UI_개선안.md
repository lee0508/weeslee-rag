# 2026-05-23 Codex 사용자 UX/UI 개선안

## 1. 작성 목적

`frontend/rag-assistant.html`을 현재의 RAG 테스트 콘솔 성격에서 문서 기반 AI Agent 화면으로 전환하기 위한 UX/UI 개선안을 정리한다.

참조 문서는 다음과 같다.

- `docs/2026-05-23_사용자html_UX_UI_개선안.md`
- `docs/2026-05-23_사용자html_UX_UI_개선안2.md`
- `docs/2026-05-23_작업우선순위.md`
- `docs/2026-05-23_작업우선순위2.md`
- `docs/2026-05-23_DesignUX_BasicRule.md`

## 2. 현재 화면 판단

현재 `rag-assistant.html`은 기능 요소는 충분하지만, 사용자가 보기에는 문서 기반 AI Agent보다 RAG 개발자 테스트 콘솔에 가깝다.

초기 화면에 `Runtime Target`, `Backend API`, `VectorDB`, `Ollama`, `Top K`, `Top Docs`, `Model`, `Temperature`, `Vector / Graph / Wiki 비율` 같은 설정이 먼저 노출된다.

사용자가 실제로 원하는 흐름은 다음이다.

```text
질문 입력
  ↓
관련 문서 검색
  ↓
추천 문서 확인
  ↓
사용할 문서 선택
  ↓
선택 문서 기반 답변 생성
  ↓
출처와 근거 확인
```

따라서 첫 개선 목표는 기능 추가가 아니라 정보 우선순위 재정렬이다.

## 3. UX 개선 핵심 방향

### 3.1 사용자 업무 중심 화면으로 전환

상단 타이틀과 설명은 기술 제품명이 아니라 업무 목적을 바로 설명해야 한다.

권장 타이틀은 다음과 같다.

```text
weeslee-rag AI Document Agent
```

권장 설명 문구는 다음과 같다.

```text
회사 문서에서 유사 사업, 제안서, RFP, 완료보고서, 기술 근거를 찾아드립니다.
```

### 3.2 검색 목적을 사용자 언어로 변경

현재 기술 중심 명칭은 사용자 업무 중심 명칭으로 바꾼다.

| 현재 명칭 | 개선 명칭 |
|---|---|
| General RAG | 일반 문서 검색 |
| Bid Project Search | 유사 사업 검색 |
| RFP Analysis | RFP 분석 |
| Graph RAG | 관계 기반 검색 |
| Knowledge Search | 지식 검색 |
| Answer Review | 답변 검토 |
| 제안서 초안 | 제안서 초안 생성 |

### 3.3 고급 설정은 접기 처리

다음 항목은 기본 화면에서 숨기고 `고급 설정` 안으로 이동한다.

- Runtime Target
- Backend API
- VectorDB
- Ollama
- Top K
- Top Docs
- Max Chunks / Doc
- Collection
- Model
- Temperature
- Vector / Graph / Wiki 비율

일반 사용자는 질문과 문서 선택에 집중하고, 관리자나 개발자만 필요할 때 펼쳐서 확인하게 한다.

## 4. 개선 후 권장 화면 구조

현재 3패널 구조는 유지하되 역할을 명확히 나눈다.

```text
┌──────────────────────────────────────────────────────────────┐
│ Header                                                       │
│ weeslee-rag AI Document Agent                                │
├────────────────┬──────────────────────────────┬──────────────┤
│ Left Panel     │ Main Panel                    │ Right Panel  │
│ 검색 범위       │ 질문 입력                      │ 선택 문서      │
│ 문서 필터       │ 추천 문서                      │ 근거 Chunk     │
│ 고급 설정       │ AI 답변                        │ 답변 생성      │
│                │ 문서 상세                      │ 제안서 초안    │
└────────────────┴──────────────────────────────┴──────────────┘
```

### 4.1 Header

상단은 서비스 목적, 검색 목적 탭, 최소 상태 정보만 보여준다.

권장 탭은 다음과 같다.

- 문서 검색
- 유사 사업 검색
- RFP 분석
- 답변 검토
- 제안서 초안 생성

`Admin`, `Live /api`, 서버 포트 정보는 일반 사용자 첫 화면에서 제거하거나 작은 상태 아이콘으로 축소한다.

### 4.2 Left Panel

좌측은 다음 3개 영역으로 정리한다.

1. 검색 범위
2. 문서 필터
3. 고급 설정

검색 범위는 다음처럼 단순화한다.

- Sample RAG
- 전체 문서
- RFP
- 제안서
- 산출물

오늘 작업 문서 기준으로 기본 검색 대상은 `weeslee_sample_rag_v1`을 우선 사용한다.

### 4.3 Main Panel

중앙은 질문 입력과 검색 결과를 담당한다.

검색창 아래에는 예시 질문 버튼을 둔다.

- 공공기관 ISP 사업 실적을 찾아줘.
- AI 관련 제안서를 찾아줘.
- OCR 구축 경험 문서를 찾아줘.
- 문서 중앙화 관련 자료를 찾아줘.
- 이 RFP와 유사한 회사 실적을 찾아줘.
- 제안서에 넣을 수 있는 수행 사례를 찾아줘.

검색 실행 후에는 추천 문서 카드와 AI 답변을 함께 보여준다.

### 4.4 Right Panel

오른쪽 패널은 선택 문서 기반 작업 패널로 정리한다.

구성 요소는 다음과 같다.

- 선택된 문서 목록
- 근거 Chunk 목록
- 선택 문서 기반 답변 생성 버튼
- 제안서 초안 생성 버튼
- Citation 목록

문서를 선택하지 않으면 답변 생성 버튼을 비활성화한다.

안내 문구는 다음과 같다.

```text
선택한 문서 내용만 근거로 답변을 생성합니다.
문서에 없는 내용은 답변하지 않습니다.
```

## 5. 검색 결과 UI 개선안

### 5.1 추천 문서 카드

추천 문서는 단순 텍스트 목록이 아니라 카드로 보여준다.

카드에 포함할 필드는 다음과 같다.

- 문서명
- 문서유형
- 문서그룹
- 세부분류
- 기관명
- 사업명
- 연도
- 유사도
- 관련 이유
- 주요 근거 Chunk
- 원문 보기
- 문서 선택

예시는 다음과 같다.

```text
추천 문서

문서명: K-water 고유 플랫폼 구축 전략계획 수립 용역 제안서
문서유형: 제안서
세부분류: 전략및방법론
기관명: 한국수자원공사
유사도: 87%

관련 이유
- 공공기관 ISP 사업이다.
- 플랫폼 구축 전략과 데이터 통합 내용이 포함되어 있다.
- 클라우드 전환 전략 근거로 활용할 수 있다.

[근거 보기] [문서 선택] [원문 보기] [제안서 문구 생성]
```

### 5.2 결과 탭 재구성

현재 결과 패널은 다음 탭으로 정리한다.

- 추천 문서
- RAG 근거
- Graph 관계
- Wiki 요약
- 관련 파일

각 탭의 역할은 다음과 같다.

| 탭 | 역할 |
|---|---|
| 추천 문서 | 최종 병합 추천 결과 |
| RAG 근거 | 실제 매칭 Chunk와 유사도 |
| Graph 관계 | 기관, 사업, 기술 관계 |
| Wiki 요약 | 사전 생성된 사업별 요약 |
| 관련 파일 | 원본 파일, 경로, 다운로드 |

## 6. RFP 분석 UX 개선안

RFP 파일 첨부는 단순 업로드가 아니라 RFP 분석 시작점으로 바꾼다.

권장 섹션명은 다음과 같다.

```text
RFP 파일로 유사 실적 찾기
```

업로드 후 실행할 작업은 체크박스로 보여준다.

- 핵심 요구사항 추출
- 기술 키워드 추출
- 유사 사업 검색
- 관련 제안서 검색
- 제안서 초안 생성

RFP 분석 결과는 다음 구조로 표시한다.

- 사업명
- 발주기관
- 사업 목적
- 주요 요구사항
- 기술 키워드
- 관련 회사 문서
- 유사 수행 실적
- 제안서 작성 포인트

## 7. 제안서 초안 UX 개선안

제안서 초안 생성은 검색 결과의 선택 문서와 반드시 연결한다.

권장 흐름은 다음과 같다.

```text
검색 또는 RFP 분석
  ↓
추천 문서 표시
  ↓
사용자 문서 선택
  ↓
제안서 섹션 선택
  ↓
선택 문서 기반 초안 생성
  ↓
출처 확인
  ↓
다운로드
```

초안 생성 버튼 근처에는 다음 문구를 표시한다.

```text
선택 문서 3건을 기반으로 제안서 초안을 생성합니다.
```

## 8. API 연동 정리 방향

UI 개선만으로 끝나지 않도록 API 응답 구조도 정리해야 한다.

검색 응답에는 `recommended_documents` 배열을 포함하는 것이 좋다.

권장 필드는 다음과 같다.

```json
{
  "document_id": "string",
  "title": "string",
  "file_name": "string",
  "document_type": "제안서",
  "document_group": "제안서",
  "document_category": "전략및방법론",
  "organization": "한국수자원공사",
  "project_name": "K-water 고유 플랫폼 구축 전략계획 수립 용역",
  "year": "2024",
  "score": 0.87,
  "reason": "공공기관 ISP와 플랫폼 구축 전략이 유사합니다.",
  "matched_chunks": [],
  "source_mode": "rag"
}
```

선택 문서 기반 답변 생성 API는 `selected_document_ids`를 필수로 받아야 한다.

답변 결과에는 `citations`를 반드시 포함한다.

Graph 결과는 UI에서 바로 렌더링 가능한 `nodes`, `edges`, `paths` 구조로 반환한다.

Wiki 결과는 `title`, `summary`, `related_documents` 구조로 반환한다.

부분 실패는 전체 실패로 처리하지 않는다.

예를 들어 RAG 검색은 성공하고 Graph 검색만 실패한 경우에는 추천 문서와 RAG 근거를 먼저 보여주고 Graph 탭에만 오류를 표시한다.

## 9. Codex 기준 우선 구현 순서

### 1순위. 화면 용어 정리

기술 용어를 사용자 업무 용어로 바꾼다.

- `Run RAG`를 `검색 실행`으로 변경한다.
- `RAG Query`를 `문서 검색 질문`으로 변경한다.
- `Recommended Documents`를 `추천 문서`로 변경한다.
- `Search Only`를 `문서 검색만`으로 변경한다.
- `Ollama Generate`를 `AI 답변 생성`으로 변경한다.

### 2순위. 고급 설정 접기

좌측 설정 중 개발자용 설정을 `고급 설정`으로 이동하고 기본 접기 상태로 둔다.

이 단계는 API 변경 없이 UI 부담을 줄일 수 있어 가장 빠르게 체감 효과가 난다.

### 3순위. 질문 중심 첫 화면 재배치

첫 화면의 중심을 상태 정보와 설정값이 아니라 질문 입력창으로 바꾼다.

예시 질문 버튼을 추가해 사용자가 바로 검색을 시작할 수 있게 한다.

### 4순위. 추천 문서 카드 개선

검색 결과를 카드형 UI로 바꾸고 문서 선택 액션을 명확히 둔다.

각 카드에는 문서명, 문서유형, 세부분류, 사업명, 유사도, 관련 이유, 근거 보기, 문서 선택 버튼을 표시한다.

### 5순위. 선택 문서 기반 답변 생성 강화

문서를 선택하지 않은 상태에서는 답변 생성을 제한한다.

선택한 문서만 근거로 답변한다는 안내 문구와 citation UI를 함께 제공한다.

### 6순위. Sample RAG 기본 연결

오늘 작업 우선순위 문서 기준으로 전체 문서 RAG보다 `weeslee_sample_rag_v1`을 먼저 기본 검색 대상으로 둔다.

샘플 검증 후 `weeslee_department_rag_v1`, `weeslee_full_rag_v1`로 확장한다.

### 7순위. RFP 분석 흐름 강화

RFP 업로드 영역을 RFP 분석 워크플로로 확장한다.

요구사항 추출, 기술 키워드 추출, 유사 문서 검색, 제안서 초안 생성 흐름을 한 화면에서 이어지게 한다.

### 8순위. Graph와 Wiki를 보조 패널로 연결

Graph RAG와 LLM Wiki는 별도 기능처럼 보이기보다 검색 결과를 보강하는 탭으로 연결한다.

사용자는 관계 그래프와 Wiki 요약을 추천 문서의 근거 보강 자료로 이해해야 한다.

### 9순위. Search Quality 피드백 연결

검색 결과 카드에 품질 피드백을 추가한다.

- 관련 있음.
- 관련 없음.
- 더 정확한 문서가 있음.
- 정답 문서로 지정.

이 피드백은 이후 Search Quality 화면에서 테스트 질문 관리와 품질 점수화로 이어진다.

## 10. 작업 시 주의사항

1. 기존 API 호출 로직은 삭제하지 않는다.
2. 먼저 UI 문구와 배치를 정리하고, 그 다음 응답 구조를 보강한다.
3. `logs/` 폴더에 작업 단계별 로그를 남긴다.
4. 인코딩 이력이 있는 큰 HTML 파일은 직접 대량 치환을 피한다.
5. 변경 전후로 inline script 파싱 검증을 수행한다.
6. 가능하면 작은 단위로 커밋한다.
7. 배포 전 로컬 검증, 서버 pull, 재시작, health check 순서를 지킨다.

## 11. 최종 결론

현재 `rag-assistant.html`은 기능은 충분하지만 UX의 중심이 문서 기반 AI Agent가 아니라 RAG 설정 콘솔에 맞춰져 있다.

가장 먼저 해야 할 일은 새 기능을 많이 추가하는 것이 아니다.

우선 화면 용어를 사용자 업무 중심으로 바꾸고, 고급 설정을 접고, 질문 입력과 추천 문서 선택 흐름을 화면 중심에 배치해야 한다.

그 다음 선택 문서 기반 답변 생성과 citation을 강화하면 `rag-assistant.html`은 문서 검색 도구에서 신뢰 가능한 문서 기반 AI Agent로 전환될 수 있다.
