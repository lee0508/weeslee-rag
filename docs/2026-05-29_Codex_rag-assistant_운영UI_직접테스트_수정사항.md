## 6. 확인된 문제

### 6.1 파일 카드 클릭 가능 영역이 불명확함

결과 파일 카드는 시각적으로 카드 전체가 클릭 가능한 것처럼 보인다.

하지만 실제 동작은 카드 헤더와 `원문` 버튼에만 연결되어 있고, 카드 본문 클릭은 아무 동작이 없다.

Lee 개선안의 “LLM이 찾은 관련 파일 영역을 카드 형태로 정리하고 보기/다운로드/경로 복사 액션 버튼을 명확히 제공” 항목과 직접 연결되는 문제다.

수정 방향은 다음이다.

- 카드 전체 클릭 시 문서 상세 패널을 열도록 한다.
- 버튼 클릭은 `event.stopPropagation()`으로 카드 클릭과 충돌하지 않게 유지한다.
- 카드에 hover 상태와 포커스 스타일을 추가한다.

### 6.2 `원문` 버튼 이름과 실제 동작이 다름

결과 카드의 `원문` 버튼은 원문 파일 미리보기를 바로 여는 것이 아니라 문서 상세 패널의 `original` 탭을 연다.

상세 패널 안에는 다시 `파일 열기` 버튼이 있어 사용자는 두 번 클릭해야 한다.

수정 방향은 다음이다.

- 버튼명을 `상세` 또는 `상세 보기`로 바꾼다.
- 원문을 바로 열려면 카드 액션에 `파일 보기` 버튼을 별도로 둔다.
- `원문`, `요약`, `근거` 탭과 `파일 보기` 액션의 역할을 분리한다.

### 6.3 상세 패널 하단 액션 버튼 접근성이 낮음

상세 패널의 `파일 열기`, `다운로드`, `경로 복사` 버튼은 패널 하단 액션 섹션에 있다.

자동화 클릭에서는 `파일 열기` 버튼이 visible 상태임에도 viewport 밖 요소로 판정됐다.

실사용에서도 패널 높이와 스크롤 위치에 따라 주요 액션을 찾기 어렵거나 클릭하기 어려울 수 있다.

수정 방향은 다음이다.

- 상세 패널 헤더 오른쪽에 `파일 보기`, `다운로드`, `경로 복사` 아이콘 버튼을 고정 배치한다.
- 하단 액션 섹션은 보조 위치로 남기거나 제거한다.
- 패널 본문은 독립 스크롤 영역으로 두고 액션 바는 sticky 처리한다.

### 6.4 미리보기 모달이 `불러오는 중...` 상태에 머물 수 있음

우측 답변 패널의 `보기` 버튼 클릭 시 미리보기 모달이 열렸지만, 테스트 시점에는 `불러오는 중...`만 표시됐다.

직접 API를 확인하면 해당 문서 상세 API는 200을 반환하지만 응답 크기가 크다.

| API | 결과 |
|---|---|
| `/api/documents/114698` | 200, 약 981KB JSON |
| `/api/documents/114698/text` | 200, 약 489KB JSON |
| `/api/documents/114698/download?format=original` | 200, 약 41.5MB PPTX |

문서 상세 API가 `raw_text`까지 포함한 큰 JSON을 내려주면 미리보기 시작이 느려질 수 있다.

수정 방향은 다음이다.

- `/api/documents/{id}`는 미리보기용 metadata와 `available_formats`만 빠르게 반환한다.
- 원문 텍스트는 `/text`, HTML은 `/html`, Markdown은 `/markdown`에서 지연 로드한다.
- 미리보기 모달에 로딩 timeout과 재시도 버튼을 추가한다.
- 로딩 중 다운로드 버튼은 먼저 활성화하되, preview body에는 “텍스트 추출본을 불러오는 중”처럼 구체 상태를 표시한다.

### 6.5 메타데이터 배지가 붙어서 보임

결과 카드 메타데이터가 다음처럼 붙어서 보인다.

```text
산출물deliverablerag_source산출물/target_model
```

사용자는 카테고리, 컬렉션, 문서 그룹, 섹션을 구분하기 어렵다.

수정 방향은 다음이다.

- 칩 사이 gap을 보장한다.
- 내부 키 `deliverable`, `rag_source`, `target_model`은 사용자 표시명으로 매핑한다.
- 표시 예시는 `산출물 · 목표모델 · rag_source`처럼 구분자를 둔다.

### 6.6 검색 점수가 모두 100%로 표시됨

상위 5개 카드가 모두 `100%`로 표시됐다.

실제 상세 패널에서는 `Best Score: 1.0000`, `Ranking Score: 9.1500`처럼 표시된다.

사용자 입장에서는 결과 간 품질 차이를 판단하기 어렵다.

수정 방향은 다음이다.

- 유사도 점수와 ranking score를 분리 표시한다.
- 100%로 고정되는 값이면 퍼센트 배지를 숨기고 `Rank #1`, `근거 N개`를 표시한다.
- 점수 표시 정책을 백엔드 응답 스키마와 맞춘다.

### 6.7 우측 답변 패널의 답변 상태가 혼재됨

검색 완료 후 우측 패널에는 `생성 중..` 텍스트와 `LLM이 찾은 관련 파일` 영역이 함께 보였다.

문서 검색은 완료됐지만 LLM 답변 생성은 별도 상태로 남아 있는 것으로 보인다.

수정 방향은 다음이다.

- 검색 완료와 답변 생성 완료 상태를 분리한다.
- 답변 생성 중이면 관련 파일 영역 위에 별도 로딩 배너를 표시한다.
- 답변 생성 실패 시 관련 파일은 유지하고 답변 영역에 실패 메시지를 표시한다.

### 6.8 사용자 화면에서 admin API 401이 발생함

콘솔 로그에 다음 요청이 보였다.

```text
GET /weeslee-rag/api/admin/faiss/category-status
401 Unauthorized
```

사용자 페이지에서 관리자 API를 호출하는 것은 불필요한 오류 로그를 만든다.

수정 방향은 다음이다.

- 사용자 페이지에서는 admin API를 호출하지 않는다.
- 필요한 상태 정보는 공개 health endpoint 또는 사용자용 status endpoint로 분리한다.
- 401은 콘솔 오류 대신 조용히 무시하거나 “권한 필요” 상태로 처리한다.

## 9. 결론

운영 화면에서 검색 자체는 정상 동작했다.

하지만 사용자가 결과 파일을 클릭해 근거를 확인하는 핵심 흐름은 아직 명확하지 않다.

가장 먼저 고칠 부분은 결과 카드 전체 클릭 동작, `원문`과 `파일 보기` 액션의 역할 분리, 미리보기 모달의 로딩 상태 처리다.

Lee 개선안의 전체 레이아웃 개선도 필요하지만, 이번 테스트 기준으로는 결과 파일 클릭과 미리보기 안정화가 우선이다.
