# 2026-05-27 Codex RAG 근거자료 데이터셋 수정 권고

## 목적

`rag-assistant.html` 사용자 페이지는 Prompt 질문에 대해 관련 문서를 보여 주는 것에서 끝나면 안 된다. 사용자가 왜 해당 문서가 검색 결과로 나왔는지 검증할 수 있도록 파일 원문, 요약, 검색 chunk, GraphRAG 관계를 함께 제공해야 한다.

따라서 `admin.html`의 Dataset Builder와 RAG Source 메뉴는 폴더 탐색용 메뉴가 아니라 사용자 답변 검증을 위한 근거 자료 생성 메뉴로 정리해야 한다.

## 현재 확인한 구조

서버의 `Document Source` 기본값 `00. RAG 소스`는 다음 구조다.

```text
00. RAG 소스
  01. RFP                         47 files
  02. 제안서                      147 files
    01. 전략및방법론              36 files
    02. 기술및기능                36 files
    03. 프로젝트관리              34 files
    04. 프로젝트지원              32 files
    05. 연구과제                   9 files
    06. 감리                       0 files
    07. PMO                        0 files
    08. PoC                        0 files
  03. 산출물                       56 files
    01. 환경분석                   8 files
    02. 현황분석                  15 files
    03. 목표모델                  19 files
    04. 이행계획                  10 files
    05. 연구과제                   4 files
    06. 감리                       0 files
    07. PMO                        0 files
    08. PoC                        0 files
```

백엔드 메타데이터 규칙은 `연구과제`, `감리`, `PMO`, `PoC`까지 이미 지원한다. 반면 `admin.html`의 RAG Source 트리와 Collection Template은 일부 항목이 빠져 있어 실제 폴더 구조와 완전히 일치하지 않는다.

## 우선순위 1

### 사용자 근거 확인 흐름을 먼저 완성한다.

사용자 페이지에서 검색 결과 문서가 표시될 때 다음 근거를 한 화면 흐름으로 확인할 수 있어야 한다.

- 파일 근거.
- 파일명, 문서 유형, 원본 경로, source_id, document_id.
- 내용 근거.
- OCR 추출 텍스트, Markdown, HTML, 요약, 검색에 매칭된 chunk.
- 관계 근거.
- 같은 프로젝트, 같은 RFP, 같은 제안서 섹션, 같은 산출물 섹션, 같은 기술, 같은 방법론, 같은 기관.
- Graph 근거.
- 검색 결과 문서 중심 GraphRAG 노드와 엣지.

권고 작업은 다음이다.

- `rag-assistant.html` 문서 카드에서 원문 보기, 요약 보기, 검색 chunk 보기, Graph 보기 이동을 명확히 분리한다.
- 검색 결과 응답에 `document_id`, `source_id`, `project_name`, `document_group`, `proposal_section`, `deliverable_section`, `evidence_snippets`, `summary_available` 여부를 일관되게 포함한다.
- 문서 상세 모달에서 `html`, `markdown`, `text`, `summary` 탭을 같은 방식으로 제공한다.
- Graph 탭은 이미 수정한 `document_id` 기반 그래프를 유지하되, 엣지 라벨을 사용자에게 이해되는 근거 문구로 표시한다.

## 우선순위 2

### Dataset Builder 산출물을 근거 자료 기준으로 명확히 분리한다.

현재 단계 구성은 적합하다. 다만 각 단계의 산출 목적을 UI와 로그에서 더 명확히 해야 한다.

권고 기준은 다음이다.

| 단계 | 현재 의미 | 근거 자료 관점 |
|---|---|---|
| 5 OCR 작업 + 청킹 시작 | 텍스트 추출과 chunk 생성 | 문서 원문, HTML, Markdown, text, 검색 chunk 근거 생성 |
| 6 임베딩 + Vector + FAISS 진행 | 검색 인덱스 생성 | Prompt와 관련 문서 및 chunk를 찾는 검색 근거 생성 |
| 7 Graph RAG 생성 | 관계 그래프 생성 | 결과 문서들이 왜 연결되는지 검증하는 관계 근거 생성 |
| 8 LLM Wiki 생성 | Wiki 문서 생성 | 프로젝트, 기관, 기술 단위 요약 근거 생성 |

권고 작업은 다음이다.

- Step 5 완료 결과에 문서별 OCR 산출물 생성 여부를 표시한다.
- Step 6 완료 결과에 snapshot, chunk 수, 색인 문서 수를 표시한다.
- Step 7 완료 결과에 source_id별 graph node, edge, document 수를 표시한다.
- Step 8 완료 결과에 source_id별 Wiki 프로젝트 수와 실패 프로젝트를 표시한다.

## 우선순위 3

### RAG Source 메뉴를 실제 폴더 구조와 맞춘다.

`admin.html`의 RAG Source 메뉴는 현재 실제 폴더 구조 중 일부만 표시한다. 운영 기준에서는 실제 `00. RAG 소스` 폴더 구조와 1:1로 맞춰야 한다.

추가해야 할 메뉴 항목은 다음이다.

- 제안서/감리.
- 제안서/PMO.
- 제안서/PoC.
- 산출물/감리.
- 산출물/PMO.
- 산출물/PoC.

권고 작업은 다음이다.

- 하드코딩된 RAG Source 트리를 실제 폴더 스캔 결과 또는 백엔드 분류 규칙에서 생성한다.
- 파일 수가 0인 폴더도 표시하되 `0` 배지를 붙인다.
- 필터 키는 백엔드 메타데이터의 `proposal_section`, `deliverable_section`, `category`, `sub_group_key`와 일치시킨다.
- 현재 `proposal_tech` 같은 UI 키와 백엔드 `proposal_technology` 같은 category 키 차이를 정리한다.

## 우선순위 4

### Collection Template을 실제 분류 체계와 맞춘다.

현재 Collection Template은 `RFP`, `제안서`, `산출물`, 그리고 일부 제안서/산출물 하위 섹션만 포함한다. 실제 구조와 백엔드 규칙을 기준으로 빠진 컬렉션을 추가해야 한다.

추가 권고 컬렉션은 다음이다.

- `rag_source_proposal_research`.
- `rag_source_proposal_audit`.
- `rag_source_proposal_pmo`.
- `rag_source_proposal_poc`.
- `rag_source_deliverable_research`.
- `rag_source_deliverable_audit`.
- `rag_source_deliverable_pmo`.
- `rag_source_deliverable_poc`.

단, 검색 품질 관점에서는 컬렉션을 너무 많이 물리적으로 나누기보다 하나의 FAISS snapshot에 메타데이터 필터를 적용하는 방식이 더 적합하다. 따라서 Collection Template은 논리 필터와 화면 구성을 위한 용도로 유지하는 것이 좋다.

## 우선순위 5

### Graph와 Graphify 표현을 근거 검증 형식으로 정리한다.

사용자가 원하는 Graph는 전체 Knowledge Graph 탐색이 아니라 검색 결과 검증용 Graph다. 따라서 Graph 탭과 Graphify 표현은 다음 기준으로 구분한다.

- Graph 탭.
- 검색 결과 문서들을 중심으로 프로젝트, 문서 그룹, 섹션, 기술, 방법론 관계를 시각화한다.
- Graphify 표현.
- 답변 근거를 문장형 또는 카드형 관계 설명으로 보여 준다.

예시는 다음이다.

```text
이 문서는 같은 프로젝트의 RFP와 연결되어 있습니다.
이 제안서의 기술및기능 섹션은 검색어의 기술 키워드와 매칭되었습니다.
이 산출물의 목표모델 문서는 동일 프로젝트의 제안서 전략및방법론과 연결됩니다.
```

권고 작업은 다음이다.

- GraphRAG 엣지의 `relation` 값을 사용자 친화적인 한국어 라벨로 변환한다.
- 검색 결과 문서별로 `근거 관계` 요약을 생성한다.
- Graph 탭에는 시각화, 문서 카드에는 짧은 관계 설명을 제공한다.

## 우선순위 6

### 데이터셋 생성 메뉴의 이름을 목적 중심으로 바꾼다.

현재 메뉴명은 기술 처리 단계 중심이다. 사용자 검증 목적을 반영하려면 다음처럼 보조 설명을 붙이는 것이 좋다.

| 현재 메뉴 | 권고 표현 |
|---|---|
| OCR 작업 + 청킹 시작 | OCR/청킹. 원문 및 검색 근거 생성 |
| 임베딩 + Vector + FAISS 진행 | 임베딩/FAISS. 관련 문서 검색 근거 생성 |
| Graph RAG 생성 | Graph RAG. 문서 관계 검증 근거 생성 |
| LLM Wiki 생성 | LLM Wiki. 프로젝트 요약 근거 생성 |

메뉴명 자체를 길게 만들기보다 단계 설명과 완료 메시지에 목적을 명시하는 방식이 좋다.

## 권고 실행 순서

1. `admin.html` RAG Source 트리와 Collection Template을 실제 폴더 구조와 일치시킨다.
2. `rag-assistant.html` 문서 상세에서 원문, 요약, 검색 chunk를 빠르게 확인하게 만든다.
3. Graph 탭의 엣지 라벨과 근거 설명을 사용자 친화적으로 바꾼다.
4. Step 5, Step 6, Step 7, Step 8 완료 메시지에 근거 산출물 통계를 표시한다.
5. Graphify 형태의 문장형 근거 요약을 문서 카드 또는 답변 하단에 추가한다.

## 결론

Dataset Builder의 단계 구성은 유지하는 것이 맞다. 수정의 핵심은 메뉴를 더 많이 만드는 것이 아니라, 각 단계의 산출물을 사용자 질문 검증용 근거로 연결하는 것이다.

`00. RAG 소스`의 폴더 구조는 RFP, 제안서, 산출물의 근거 흐름에 잘 맞는다. 다만 admin 메뉴와 Collection Template이 실제 폴더 구조를 완전히 반영하지 못하므로, 이 부분을 먼저 보완해야 한다.
