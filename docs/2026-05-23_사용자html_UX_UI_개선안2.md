# 2. 현재 UX/UI의 핵심 문제

## 2.1 개발자 중심 설정이 너무 먼저 보인다

현재 화면에는 다음 설정이 초기에 많이 노출된다.

- Runtime Target
- Backend API
- VectorDB
- Ollama
- Top K
- Top Docs
- Max Chunks / Doc
- Collection
- Vector / Graph / Wiki 비율
- Temperature
- Model

이 설정들은 중요하지만 일반 사용자의 첫 화면에 모두 보이면 어렵게 느껴진다.

일반 사용자는 다음을 먼저 원한다.

> “무엇을 찾고 싶은지 입력하고, 관련 문서를 보고, 답변을 받고 싶다.”

따라서 개발자 설정은 기본값으로 숨기고,
고급 설정 패널로 이동해야 한다.

## 2.2 검색 목적이 명확히 구분되지 않는다

현재 검색 모드는 다음처럼 구성되어 있다.

- General RAG
- Bid Project Search
- RFP Analysis
- Graph RAG

이 자체는 좋다.

하지만 사용자는 이 용어를 보고 바로 이해하기 어렵다.

사용자 중심 문구로 바꾸면 다음이 좋다.

| 현재 명칭 | 개선 명칭 |
|---|---|
| General RAG | 일반 문서 검색 |
| Bid Project Search | 유사 사업 검색 |
| RFP Analysis | RFP 분석 |
| Graph RAG | 관계 기반 검색 |
| Knowledge Search | 지식 검색 |
| Answer Review | 답변 검토 |
| 제안서 초안 | 제안서 초안 생성 |

## 2.3 “검색 → 문서 선택 → 답변 생성” 흐름이 약하다

문서 기반 AI Agent에서 가장 중요한 것은 다음 흐름이다.

```text
질문 입력
  ↓
관련 문서 검색
  ↓
사용자가 문서 선택
  ↓
선택 문서만 기반으로 답변 생성
  ↓
출처 표시

현재 화면은 검색과 답변 생성 기능이 함께 있지만,
사용자가 “어떤 문서를 근거로 답변이 생성되는지” 명확히 보기 어렵다.

따라서 검색 결과에서 문서 선택 영역을 더 강하게 만들어야 한다.

2.4 Graph / Wiki 기능이 사용자의 언어로 설명되지 않는다

현재 화면에는 Vector 100%, Graph 0%, Wiki 0% 같은 비율 설정이 보인다.

이 기능은 기술적으로는 좋지만, 사용자는 다음처럼 이해해야 한다.

Vector 검색: 문장 유사도 기반 검색
Graph 검색: 기관, 사업, 기술 관계 기반 검색
Wiki 검색: 사전에 요약된 지식 문서 검색

따라서 UI에서는 다음처럼 표시하는 것이 좋다.

검색 방식

[문장 유사도 중심]
[기관/사업/기술 관계 중심]
[요약 지식 중심]
[균형 검색]

고급 설정에서만 Vector / Graph / Wiki 비율을 조정하게 한다.

16. Claude 작업 지시서
작업 목적

현재 rag-assistant.html을 문서 기반 AI Agent UX에 맞게 개선한다.

작업 대상
frontend/rag-assistant.html
관련 CSS/JS 파일
작업 지시
1. 페이지 상단 타이틀을 “weeslee-rag AI Document Agent”로 변경한다.
2. 설명 문구를 “회사 문서에서 유사 사업, 제안서, RFP, 완료보고서, 기술 근거를 찾아드립니다.”로 변경한다.
3. 검색 모드명을 사용자 업무 중심 문구로 변경한다.
   - General RAG → 일반 문서 검색
   - Bid Project Search → 유사 사업 검색
   - RFP Analysis → RFP 분석
   - Graph RAG → 관계 기반 검색
4. 검색창 아래 예시 질문 버튼을 추가한다.
5. 좌측 설정 영역을 “검색 범위 / 문서 필터 / 고급 설정” 3개 섹션으로 정리한다.
6. 고급 설정은 기본 접기 상태로 변경한다.
7. 결과 패널을 “추천 문서 / RAG 근거 / Graph 관계 / Wiki 요약 / 관련 파일” 탭으로 재구성한다.
8. 추천 문서는 카드형 UI로 표시한다.
9. 각 문서 카드에는 문서명, 문서유형, 기관명, 연도, 유사도, 관련 이유, 버튼을 표시한다.
10. 오른쪽 패널은 “선택 문서 / 근거 Chunk / 답변 생성 / 제안서 초안” 중심으로 재구성한다.
11. 문서를 선택하지 않으면 답변 생성 버튼을 비활성화한다.
12. 답변 생성 영역에 “선택한 문서 내용만 근거로 답변을 생성합니다.” 안내 문구를 추가한다.
13. RFP 파일 첨부 영역은 “RFP 파일로 유사 실적 찾기” 섹션으로 개선한다.
14. 제안서 초안 생성은 선택 문서 기반으로 동작하도록 UI 문구를 변경한다.
15. 기존 API 호출 로직은 삭제하지 말고, UI 구조만 우선 개선한다.
16. API 미연결 상태에서도 mock 데이터로 화면 확인이 가능하게 한다.
17. Codex 작업 지시서
작업 목적

개선된 rag-assistant.html UI가 실제 문서 기반 AI Agent처럼 동작하도록 API 응답 구조와 프론트 연동을 정리한다.

작업 지시
1. 현재 rag-assistant.html에서 호출하는 API 목록을 분석한다.
2. 검색 실행 API의 응답에 recommended_documents 배열을 추가한다.
3. 각 recommended_document에 다음 필드를 포함한다.
   - document_id
   - title
   - file_name
   - document_type
   - organization
   - project_name
   - year
   - score
   - reason
   - matched_chunks
   - source_mode
4. 선택 문서 기반 답변 생성 API를 확인하거나 추가한다.
5. 문서 선택 없이 답변 생성 요청이 들어오면 에러를 반환한다.
6. 답변 생성 API는 selected_document_ids를 필수로 받는다.
7. 답변 결과에는 citations를 반드시 포함한다.
8. RFP 업로드 API와 RFP 분석 API를 분리한다.
9. RFP 분석 결과에는 requirements, keywords, recommended_documents를 포함한다.
10. 제안서 초안 생성 API는 selected_document_ids와 proposal_sections를 필수로 받는다.
11. Graph RAG 결과는 UI에서 표시 가능한 nodes, edges, paths 구조로 반환한다.
12. Wiki 결과는 title, summary, related_documents 구조로 반환한다.
13. 부분 실패 처리를 구현한다.
    - RAG 성공, Graph 실패 시 전체 실패로 처리하지 않는다.
14. 검색 세션 기록 저장 구조를 정리한다.
15. API 응답이 없을 때 fallback mock 응답을 반환하지 말고 명확한 error message를 반환한다.
18. 최종 결론

현재 rag-assistant.html은 기능적으로는 많은 요소를 가지고 있지만,
사용자 입장에서는 기술 설정이 너무 앞에 나와 있어 어렵게 보인다.

따라서 개선 방향은 다음과 같다.

기술 중심 RAG 콘솔
  ↓
문서 기반 AI Agent
  ↓
업무 목적 중심 검색 화면
  ↓
추천 문서 선택
  ↓
선택 문서 기반 답변 생성
  ↓
출처 기반 제안서 초안 생성

가장 먼저 해야 할 작업은 전체 기능 추가가 아니라 다음 3가지다.

화면 용어를 사용자 업무 중심으로 변경
고급 설정을 접고 검색창과 추천 문서를 중심으로 재배치
선택 문서 기반 답변 생성 흐름을 명확히 표시

이 3가지만 적용해도 현재 페이지는 RAG 테스트 화면에서
문서 기반 AI Agent 화면으로 크게 개선된다.


핵심 결론은 **지금 화면은 기능은 충분하지만, UX의 중심이 “AI Agent가 문서를 찾아주는 업무 흐름”이 아니라 “RAG 설정 콘솔”에 맞춰져 있다는 점**입니다. 먼저 문구, 배치, 문서 선택 흐름만 개선해도 사용성은 크게 좋아집니다.
