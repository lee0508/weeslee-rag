## 0. 핵심 원칙 (데이타셋2.md 기준)

### 원칙 1. 기존 API 삭제 금지

기존 API는 그대로 유지합니다.

| 유지할 API | 설명 |
|-----------|------|
| /api/rag/query | RAG 검색 |
| /api/rag/search | RAG 검색 |
| /api/rag/answer | RAG 답변 생성 |
| /api/files/download | 파일 다운로드 |
| /api/files/view | 파일 보기 |
| /api/files/text | 텍스트 조회 |
| /api/wiki/projects | Wiki 목록 |
| /api/graph/summary | Graph 요약 |
| /api/admin/faiss/status | FAISS 상태 |

새 기능은 아래처럼 추가 API로 구현합니다.

| 추가할 API | 설명 |
|-----------|------|
| /api/rag/similar-files | 유사 파일 검색 |
| /api/rag-source/metadata/build | RAG 소스 메타데이터 생성 |
| /api/rag-source/graph/build | RAG 소스 그래프 생성 |
| /api/rag-source/status | RAG 소스 상태 조회 |

### 원칙 2. 기존 Collection 유지 + 새 Collection 추가

기존 Collection은 삭제하지 않습니다. 00. RAG 소스 전용 Collection만 추가합니다.

### 원칙 3. 기존 Metadata 구조 유지 + 필드 추가

## 6. GraphDB 노드/엣지 설계

### 6.1 노드 유형

```
(:Project {name, type, organization, year})
(:RFP {file_name, source_path, ...})
(:ProposalSection {section_type, file_name, ...})
(:DeliverableSection {section_type, file_name, ...})
(:Technology {name})
(:BusinessDomain {name})
(:Chunk {content, embedding_id})
```

### 6.2 관계 유형

```
(:Project)-[:HAS_RFP]->(:RFP)
(:Project)-[:HAS_PROPOSAL_SECTION]->(:ProposalSection)
(:Project)-[:HAS_DELIVERABLE_SECTION]->(:DeliverableSection)

(:ProposalSection)-[:RESPONDS_TO]->(:RFP)
(:DeliverableSection)-[:IMPLEMENTS]->(:RFP)
(:DeliverableSection)-[:BASED_ON]->(:ProposalSection)

(:Project)-[:USES_TECHNOLOGY]->(:Technology)
(:Project)-[:IN_DOMAIN]->(:BusinessDomain)
```
