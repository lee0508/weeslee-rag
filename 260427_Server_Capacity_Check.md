# 2026-04-27 서버 용량 / 자원 점검

## 1. 점검 대상

- 서버: `192.168.0.207`
- 프로젝트 경로: `/data/weeslee/weeslee-rag`

## 2. 디스크 상태

### 2.1 루트 파티션 `/`

- 크기: `98G`
- 사용: `86G`
- 여유: `7.8G`
- 사용률: `92%`

판단:

- 루트 파티션은 여유가 적다.
- 신규 패키지 설치, 로그 증가, 시스템 레벨 캐시 증가는 주의가 필요하다.

### 2.2 데이터 파티션 `/data`

- 크기: `814G`
- 사용: `285G`
- 여유: `489G`
- 사용률: `37%`

판단:

- 프로젝트 원본 snapshot, 추출 산출물, 인덱스 저장에는 충분하다.
- 앞으로도 문서 자산은 반드시 `/data/weeslee/weeslee-rag` 아래에 유지하는 것이 맞다.

## 3. inode 상태

- `/`: `8%`
- `/data`: `2%`

판단:

- inode 부족 문제는 현재 없다.

## 4. 메모리 상태

- 총 메모리: `61Gi`
- 사용: `9.3Gi`
- 가용: `52Gi`
- swap: `8.0Gi` 중 `91Mi` 사용

판단:

- 메모리는 충분하다.
- FAISS, Ollama, chunk 처리, RAG 조립 실험을 이어가기에 무리가 없다.

## 5. 현재 프로젝트 사용량

- `/data/weeslee/weeslee-rag`: `609M`
- `data/raw`: `314M`
- `data/staged`: `8.9M`
- `data/indexes`: `14M`

판단:

- 현재 1차 PoC 데이터는 매우 작은 수준이다.
- 이후 수백~수천 문서로 늘어나도 `/data` 기준으로는 여유가 크다.

## 6. 개발 진행 관점 결론

### 6.1 지금 가능한 것

1. snapshot 확대
2. 추출/OCR 확장
3. FAISS 인덱스 확장
4. RAG 조립 고도화

### 6.2 주의할 것

1. 시스템 패키지 설치는 루트 파티션 여유를 보며 진행
2. 대형 캐시나 임시 파일은 `/tmp`보다 `/data` 하위 작업 디렉터리 활용 고려
3. 로그/venv/모델 캐시가 `/` 쪽에 쌓이지 않도록 점검 필요

## 7. 최종 판단

- `프로젝트 데이터 저장 여유`: 충분
