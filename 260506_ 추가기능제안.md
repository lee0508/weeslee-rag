해당 기능 제안에 대한 분석과 결론 내용 입니다.
아래 내용을 분석해서 현재 작업 완료된 내용에서 기능을 추가할 수 있는 방안을 제시해 주세요.

현재 위즐리앤컴퍼니 문서 중앙화 시스템에 Andrej Karpathy 의 “LLM Wiki” 패턴을 접목하는 것은 매우 좋은 방향입니다. 다만 중요한 점은:

"RAG를 버리고 LLM-Wiki로 간다"
❌

"현재 RAG 시스템 위에
LLM-Wiki 계층을 추가한다"
⭕

입니다.

Karpathy의 핵심 아이디어는:

지식을 매 질의마다 다시 추출하지 말고,
LLM이 지속적으로 관리하는
구조화된 위키(knowledge artifact)를 만든다.

입니다.

그리고 현재 당신의 프로젝트 구조는 이미 상당 부분이 LLM-Wiki와 잘 맞습니다.

1. 현재 프로젝트와 LLM-Wiki의 구조적 유사성

현재 구조:

원본 문서
→ snapshot
→ metadata
→ chunk
→ FAISS
→ RAG
→ graph

Karpathy LLM-Wiki 구조:

raw/
→ wiki/
→ schema/
→ index.md
→ entity pages
→ cross references

즉 현재 프로젝트는:

RAG 기반 문서 중앙화

이지만,

조금만 확장하면:

Persistent Knowledge System
(지속적 지식 축적 시스템)

으로 진화할 수 있습니다.

2. 현재 구조에서 가장 잘 맞는 부분

당신의 시스템은 이미 아래 개념을 가지고 있습니다.

(1) metadata

현재:

{
  "project_name": "...",
  "organization": "...",
  "document_type": "...",
  "keywords": [...]
}

LLM-Wiki에서는 이것이:

Entity Page
Concept Page
Project Page

로 발전합니다.

(2) graph schema

이미 문서에서:

Document
Project
Organization
Requirement
Methodology
Deliverable

노드를 정의했습니다.

이건 사실상:

LLM-Wiki의 backbone

입니다.

(3) provenance

현재 이미:

EXTRACTED
INFERRED
AMBIGUOUS

개념이 있습니다.

이건 Karpathy 스타일에서 매우 중요합니다.

왜냐하면 LLM-Wiki는:

LLM hallucination 방지

를 위해:

"이 정보가 어디서 왔는가?"

를 계속 유지해야 하기 때문입니다.

3. 가장 중요한 판단

현재 프로젝트는:

RAG만으로 끝내기에는
너무 좋은 구조

입니다.

왜냐하면 현재 문서 중앙화 시스템의 실제 목적은:

"문서 검색"

이 아니라,

"컨설팅 지식 축적"

이기 때문입니다.

4. 실제로 어떻게 접목해야 하는가?

여기서 가장 중요합니다.

❌ 하면 안 되는 방향
모든 chunk를 markdown wiki로 변환

이건 실패합니다.

이유:

chunk 수가 폭발
관리 불가능
LLM 비용 증가
중복 증가
⭕ 추천 구조

추천은:

RAG Layer
+
LLM-Wiki Layer

2계층 구조입니다.

5. 추천 아키텍처

현재:

documents
→ chunks
→ FAISS

추가:

wiki/
├── organizations/
├── projects/

├── document_types/
└── timelines/

7. 가장 중요한 효과

현재 RAG만 사용하면:

질문할 때마다
다시 retrieval
다시 reasoning

합니다.

하지만 LLM-Wiki가 추가되면:

이미 축적된 지식 구조를 활용

하게 됩니다.

즉:

검색
→
지식 활용

으로 바뀝니다.

8. 현재 프로젝트에서 특히 좋은 이유

위즐리 프로젝트는 일반 문서 시스템이 아닙니다.

현재 실제 목적은:

RFP 분석
→ 유사 제안서 추천
→ 초안 생성
→ 방법론 재사용
→ 조직 지식 축적

입니다.

이건 사실상:

Consulting Memory System

입니다.

Karpathy LLM-Wiki와 매우 잘 맞습니다.

9. 하지만 중요한 현실 판단

여기서 매우 중요합니다.

지금 당장:

LLM-Wiki 전체 구현

은 하면 안 됩니다.

왜냐하면 현재 가장 중요한 것은:

검색 품질

이기 때문입니다.

13. 가장 중요한 결론

현재 위즐리 문서 중앙화 프로젝트는:

"단순 RAG"

"AI 기반 컨설팅 지식 운영체계"
