맞습니다. 이 문제는 FAISS 검색 실패가 아니라 Ollama 생성 단계 timeout 문제입니다.
현재 판단:
retrieval-only 테스트 = 먼저 실행해야 함LLM 생성 테스트 = 별도/느리게 실행해야 함
1. 즉시 해결: 벤치마크를 2단계로 분리
A. retrieval-only- FAISS 검색- category filter- max_chunks_per_doc- rerank- top-k docs 확인B. generation- Ollama/gemma4 답변 생성- 소수 쿼리만 실행- timeout 길게 설정
먼저 retrieval-only로 final_report, presentation이 문서를 반환하는지 확인하세요.
2. Ollama timeout 늘리기
Python 요청 코드에서 timeout을 60초 → 180~300초로 늘립니다.
import requestsresponse = requests.post(    "http://localhost:11434/api/generate",    json={        "model": "gemma4:latest",        "prompt": prompt,        "stream": False,        "options": {            "temperature": 0.2,            "num_ctx": 4096,            "num_predict": 800        }    },    timeout=300  # 기존 60초 → 300초)
3. 동시 요청 금지
Ollama는 로컬 서버에서 동시 생성 요청이 몰리면 후반 쿼리가 timeout될 수 있습니다.
벤치마크는 일단 순차 실행하세요.
import timefor query in queries:    result = run_one_query(query)    time.sleep(2)  # Ollama 숨 고르기
가능하면 worker 수를 1로 제한합니다.
export OLLAMA_NUM_PARALLEL=1
Ollama 재시작:
pkill ollamaOLLAMA_NUM_PARALLEL=1 ollama serve
4. 생성 길이 제한
보고서/발표자료 쿼리는 context가 길어져 timeout이 잘 납니다. 테스트용은 짧게 제한하세요.
"options": {    "temperature": 0.2,    "num_ctx": 4096,    "num_predict": 500}
운영용은 나중에 num_predict=1200~2000으로 늘리면 됩니다.
5. 테스트용 모델 분리
품질 테스트는 gemma4로 하되, 벤치마크 반복 테스트는 더 가벼운 모델을 권장합니다.
retrieval 품질 테스트: LLM 없음빠른 생성 테스트: qwen2.5:7b 또는 llama3.1:8b최종 품질 테스트: gemma4
6. 가장 추천하는 실행 순서
# 1) retrieval-only 먼저python3 backend/scripts/evaluate_retrieval_only.py
# 2) Ollama 상태 확인curl http://localhost:11434/api/tags
# 3) generation benchmark는 순차 + 긴 timeoutpython3 backend/scripts/evaluate_rag_quality.py \  --timeout 300 \  --sleep 2 \  --max-concurrency 1
결론
지금은 final_report, presentation이 0 docs인 게 아니라, 검색은 됐지만 Ollama 생성 응답이 60초 안에 끝나지 않아 실패 처리된 것일 가능성이 큽니다.
따라서 다음 순서가 맞습니다.
1. retrieval-only로 검색 품질 검증
2. Ollama timeout 300초로 증가
3. 동시 요청 1개로 제한
4. 생성 토큰 수 제한
5. 최종 생성 품질 테스트만 별도 실행
