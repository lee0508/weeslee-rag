현재 시스템은
“레거시 MySQL/ChromaDB 구조”와
“신규 FAISS/JSONL RAG 구조”가 함께 존재하지만,

추출기 계층은 신규 FAISS 파이프라인에서도 충분히 재사용 가능
1. 핵심 판단

/extractors 폴더는 버리지 말고 FAISS 배치 파이프라인의 표준 추출 모듈로 가져가야 합니다.

특히 현재 이미 지원되는 형식이 좋습니다.

PDF     → pdfplumber + tesseract OCR fallback
DOCX    → python-docx
PPTX    → python-pptx
XLSX    → openpyxl
HWPX    → zip/xml 직접 추출

통합 추출기 DocumentExtractor가 PDF, DOCX, PPTX, XLSX, HWPX를 라우팅하고 있습니다.

2. 가장 중요한 장점
PDF OCR fallback이 이미 있음

PDFExtractor는 PDF가 스캔 PDF인지 검사한 뒤, OCR이 켜져 있으면 tesseract로 처리합니다.

텍스트 PDF → pdfplumber
스캔 PDF → tesseract OCR

따라서 Batch-002 실패 4건의 해결 방향은 새 개발이 아니라:

extract_manifest_batch.py가
이 PDFExtractor(use_ocr=True)를 쓰도록 연결

하는 것입니다.

3. HWPX도 이미 추출 가능

HwpxExtractor는 .hwpx를 ZIP으로 열고 Preview/PrvText.txt와 Contents/section*.xml을 읽습니다.

즉 Phase 2로 미뤘던 HWPX도 이제는 다음 배치부터 포함할 수 있습니다.

다만 HWPX는 추출 품질 검증이 필요합니다.

1차: 내용 추출 성공 여부
2차: 제목/섹션 분리 품질
3차: RFP/제안서 구분 메타데이터
4. /core 분석

config.py에는 Ollama, ChromaDB, FAISS snapshot, knowledge source, CORS 설정이 모두 들어 있습니다.

주의할 점은 faiss_snapshot 기본값이 아직 과거 top5 snapshot입니다.

faiss_snapshot: str = "snapshot_2026-04-27_batch-001-top5-v2"

따라서 운영 기준은 .env 또는 data/active_index.json을 우선해야 합니다.

database.py는 MySQL SQLAlchemy 초기화용입니다.
현재 FAISS 중심 구조에서는 필수는 아니고, 레거시/향후 이력관리용으로 두는 것이 좋습니다.

5. 개발 방향 정리
유지할 것
/core/config.py
/core/database.py
/extractors/*

단, 역할을 재정의합니다.

config.py        → 환경설정 유지
database.py      → 현재는 optional, 향후 이력/사용자 관리용
extractors       → FAISS ingest pipeline의 표준 추출기
6. 바로 해야 할 작업
1순위: extract_manifest_batch.py와 DocumentExtractor 통합

현재 배치 추출 스크립트가 별도 로직이면 정리해야 합니다.

목표:

from app.extractors.extractor import DocumentExtractor

extractor = DocumentExtractor(use_ocr=args.use_ocr)
result = await extractor.extract(file_path)

이렇게 통일해야 합니다.

7. OCR 처리 방향

Batch 실패 문서를 재처리하기 위해서는 다음이 필요합니다.

tesseract --version
python3 -c "import pytesseract, pdf2image; print('ok')"

그리고 서버에 없으면:

sudo apt install -y tesseract-ocr tesseract-ocr-kor poppler-utils
pip install pytesseract pdf2image

단, sudo 권한이 없으면 관리자에게 설치 요청해야 합니다.

8. HWPX 처리 방향

다음 배치부터 .hwpx도 포함 가능하게 바꿔야 합니다.

기존:
.pdf, .pptx, .docx, .xlsx

변경:
.pdf, .pptx, .docx, .xlsx, .hwpx

HWPX 추출기는 이미 존재합니다.

9. category 추출 보완

현재 extractors는 텍스트만 추출합니다.
RAG 품질에는 아래 metadata가 더 중요합니다.

document_id
category
project_name
organization
folder_year
source_path
extension
extraction_method
is_scanned
content_length
confidence

따라서 extract result metadata에 다음을 추가해야 합니다.

{
  "extraction_method": "pdfplumber",
  "is_scanned": false,
  "content_length": 12345,
  "metadata_confidence": {
    "project_name": 0.85,
    "category": 0.9,
    "organization": 0.6
  }
}

11. 다음 실행 문서에 넣어야 할 핵심 내용
1. MySQL/ChromaDB는 현재 운영 검색 경로에서 제외
2. FAISS + JSONL을 운영 기준으로 확정
3. extractors는 신규 FAISS ingest 파이프라인에서 재사용
4. PDF OCR fallback 활성화
5. HWPX 추출 배치 포함
6. active_index.json 기준으로 검색 인덱스 통일
7. Graphify는 metadata 기반 1차 rule-based로 구현
최종 결론

이번 /core, /extractors 파일은 매우 중요합니다.

특히 결론은 이것입니다.

extractors는 레거시가 아니라 핵심 자산입니다.
FAISS 파이프라인과 연결해서
PDF/OCR/HWPX/DOCX/PPTX/XLSX 추출 표준 계층으로 사용해야 합니다.

다음 개발 방향은 추출기 표준화 → OCR 재처리 → HWPX 포함 → FAISS 재인덱싱 → 사업명 검색 고도화 순서가 가장 좋습니다.
