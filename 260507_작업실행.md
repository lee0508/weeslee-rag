핵심 방향은 이렇게 잡는 것이 좋습니다.

1. 먼저 검색 품질을 안정화한다.
2. 그 다음 운영 관리자 기능을 정리한다.
3. 이후 LLM-Wiki / Graphify / Incremental Indexing으로 확장한다.
1순위 — 오늘 바로 해야 할 작업
1. Category Sub-Index 빌드

최우선입니다.

이유는 현재 Bid Project Search, RFP Analysis, Knowledge Search가 모두 카테고리 필터 품질에 의존하기 때문입니다.

cd /data/weeslee/weeslee-rag

nohup .venv/bin/python3 backend/scripts/build_category_indexes.py \
  --combined-chunks data/staged/chunks/snapshot_2026-05-06_combined-v1_chunks.jsonl \
  --output-dir data/indexes/faiss \
  --snapshot snapshot_2026-05-06_combined-v1 \
  --embedding-provider ollama \
  --ollama-model nomic-embed-text \
  </dev/null > /tmp/category_index.log 2>&1 &

확인 명령어:

tail -f /tmp/category_index.log

완료 후 확인:

ls -lh data/indexes/faiss/*rfp* *proposal* *kickoff* *final_report* *presentation*
2. Category 검색 검증

서브 인덱스 생성 후 바로 테스트해야 합니다.

테스트 항목:
1. rfp 검색
2. proposal 검색
3. kickoff 검색
4. final_report 검색
5. presentation 검색

특히 기존 약점이었던 RFP 검색이 개선되는지 확인해야 합니다.

3. OCR 환경 확인

이건 두 번째 우선순위입니다.

tesseract --version
python3 -c "import pytesseract, pdf2image; print('OCR OK')"

2순위 — 5/8~5/9 작업
4. Benchmark Automation

이제부터는 수동 테스트가 아니라 자동 평가로 가야 합니다.

구조:

tests/
└── queries/
    ├── rfp.json
    ├── proposal.json
    ├── kickoff.json
    ├── final_report.json
    ├── presentation.json
    └── bid_project.json

평가 모드는 반드시:

search_only

로 고정하세요.

Ollama 생성까지 포함하면 timeout 때문에 검색 품질 평가가 흐려집니다.

8. Incremental Indexing

이건 중요하지만 지금 당장은 후순위입니다.
