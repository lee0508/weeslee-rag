2. 현재 핵심 API 구조
실제 사용자 검색

rag.py가 현재 핵심입니다. 활성 인덱스는 data/active_index.json에서 읽고, category가 있으면 category sub-index를 우선 사용합니다.

POST /api/rag/query

현재 구조는 좋습니다.

다음에 추가할 것은:

mode=bid_project
query_expansion=true
rerank_profile=isp_ai

입니다.

3. 관리자 콘솔

faiss_admin.py는 현재 관리자 콘솔의 중심입니다.

이미 다음 기능이 있습니다.

GET  /api/admin/faiss/status
GET  /api/admin/faiss/indexes
POST /api/admin/faiss/jobs
GET  /api/admin/faiss/jobs
GET  /api/admin/faiss/jobs/{job_id}/stream
GET  /api/admin/faiss/category-status
POST /api/admin/faiss/benchmark
POST /api/admin/faiss/activate

이 파일은 그대로 중심으로 두면 됩니다.

단, category 목록이 현재:

_CATEGORIES = ["ISP", "ISMP", "ODA", "정책연구"]

로 되어 있습니다.

하지만 현재 RAG Assistant category는:

rfp
proposal
kickoff
final_report
presentation

입니다.

이 부분은 반드시 통일해야 합니다.

4. Graphify API

graph.py는 현재 Graphify 1차 구현으로 적합합니다. JSONL 파일을 읽어서 summary, project, document 단위 API를 제공합니다.

GET  /api/graph/summary
GET  /api/graph/projects
GET  /api/graph/project/{name}
GET  /api/graph/document/{doc_id}
POST /api/graph/build

개발 방향은 좋습니다.

다음 보완:

1. organization 노드 추가는 2차
2. relation confidence 추가
3. RAG 검색 결과에서 graph link 표시
5. 파일 다운로드 API

files.py는 중요합니다. 검색 결과 문서의 source_path를 받아 실제 파일을 다운로드하게 합니다. Linux 절대경로, Windows 드라이브, UNC, 상대경로를 모두 처리합니다.

이건 현재 시스템의 실사용성을 높이는 핵심 API입니다.

유지하세요.

6. 상태 체크 API

health.py는 API, Ollama, FAISS 상태를 확인합니다.

다만 active index 경로가 주의 필요합니다.

health.py는:

_ACTIVE_INDEX_PATH = _PROJECT_ROOT / "data" / "active_index.json"

rag.py도 같은 경로를 사용합니다.

반면 우리가 이전에 논의한 위치는:

data/indexes/faiss/active_index.json

였습니다.

현재 코드 기준으로는:

data/active_index.json

으로 통일하는 것이 좋습니다.

7. OCR API

ocr.py는 기능은 있으나, 현재 운영 파이프라인과 직접 연결되어 있지는 않습니다.

현재 제공:

POST /api/ocr/process-pdf
POST /api/ocr/process-image
POST /api/ocr/extract
POST /api/ocr/smart-extract

다음 작업:

스캔 PDF 실패 4건을 처리할 수 있도록
FAISS pipeline의 extract_manifest_batch.py에서
--use-ocr 옵션과 연결

즉 OCR API 자체보다 배치 파이프라인 연결이 더 중요합니다.

8. 레거시로 분리할 API

다음 3개는 지금 핵심 개발에서 제외하는 것이 좋습니다.

admin.py
collections.py
documents.py

이유:

admin.py는 document_pipeline_service, MySQL, ChromaDB 기반입니다.
collections.py도 MySQL Collection + VectorDBService 중심입니다.
documents.py도 DB Document 모델과 업로드 저장소 중심입니다.

현재 실제 운영 중인 RAG는 FAISS + JSONL이므로, 이 3개를 섞으면 구조가 복잡해집니다.
